# Unity 2D 도트 액션 플랫포머 (입문·팀용)

유니티로 2D 플랫포머를 **이 저장소 문서만** 따라 하면 **PC 빌드까지** 이어갈 수 있게 정리했습니다. **씬·게임 흐름·씬 전환·씬별 프리팹**은 **[`#scene-flow`](#scene-flow)** + [04](docs/04-요구사항-및-시스템설계.md)·[02](docs/02-준비사항-체크리스트.md)·[07 Phase 0](docs/07-단계별-개발-가이드.md)에 녹아 있습니다. **씬 흐름 인쇄**: 이 README **`#scene-flow`** 절. [32](docs/32-Unity-씬-게임흐름-씬전환-가이드.md)는 링크 모음. **03-게임기획서-초안·체크리스트(29)·데이터 매핑(24)·맵·UI·병렬 개발**까지 같은 **통합 가이드** 안에 포함되어 있습니다.  
**단계·준비 확인**: 별도 집중 문서 없이 **각 문서 말미의 「단계 연결·준비 확인」** 과 **[07](docs/07-단계별-개발-가이드.md)** 의 Phase별 **시작 전/완료 후** 를 따릅니다.  
코드를 **세 폴더**로 나누는 규칙(`_Framework` · `_Content` · `_UI`)은 **아래 표가 정본**입니다. (옛말 **A/B/C** 대신 **폴더 이름·기반/컨텐츠/UI** 로 부릅니다.) [02](docs/02-준비사항-체크리스트.md)·[04](docs/04-요구사항-및-시스템설계.md)·[24](docs/24-기획-데이터-코드-연결-템플릿.md)·[07](docs/07-단계별-개발-가이드.md)·[10](docs/10-UI-HUD-디자인-및-구현-가이드.md)·**[#map-asset-pipeline](#map-asset-pipeline)**·[28](docs/28-기획서-맵구성-에셋-구현-연동.md)(맵 심화)에도 반복됩니다. **게임 구조 설계 → 코어 → 컨텐츠 → UI** 는 **[네 축](#dev-flow-axes)**. **트리거·인터랙션·이벤트 채널** 은 **[필요 시 순서](#interaction-events)**. **프레임워크·컨텐츠·UI** 체크리스트·`GameEvents` 샘플·경계 예시는 **아래 [`#folder-depth`](#folder-depth)** + [07](docs/07-단계별-개발-가이드.md) + [04](docs/04-요구사항-및-시스템설계.md) + [10](docs/10-UI-HUD-디자인-및-구현-가이드.md)에 통합되어 있습니다. **한 파일로만** 보려면 [31](docs/31-프레임워크-컨텐츠-UI-구분-및-구현.md). 실습 허브는 **[07](docs/07-단계별-개발-가이드.md)**, 코드 복사는 [26](docs/26-CSharp-코드복사-부록.md)(선택).

## 세 폴더로 나누기 — `_Framework` · `_Content` · `_UI`

**A/B/C 같은 알파벳은 쓰지 않습니다.** 아래 **폴더 이름**과 **한글 이름(기반·컨텐츠·UI)** 만 쓰면 됩니다. **9b(G10)** = 이 표를 팀에 한 번 맞춘 것.

| 부를 이름 | 폴더 | 무엇을 두나 | 하면 안 되는 것 |
|-----------|------|-------------|-----------------|
| **기반** (프레임워크) | `_Framework/` | 씬 전환, 런 상태, **이벤트**(HP·게임오버 등), 전투 **규칙 훅** | 특정 스테이지명 하드코딩, 보스 수치를 코드 상수로, UI 문구·Canvas 직접 조작 |
| **컨텐츠** | `_Content/` | **SO**·스테이지 씬·맵·적·증강 **프리팹**·장애물 배치 | `FindObjectOfType<UIManager>` 등 UI 직접 찾기 |
| **UI** | `_UI/` | Canvas·HUD·패널·버튼, **표시·입력만** | `currentHp -= damage`, 스테이지 클리어 판정, 데미지 공식 |

**연결**: **UI → 기반**(이벤트·세션만), **컨텐츠 → 기반**(데이터 주입), **UI ↔ 컨텐츠 직접 참조 금지**.

### 세 폴더 심화 — 권장 트리·체크리스트·GameEvents (G10·구현 전체)

**분산 정본**: 같은 내용이 [07 Phase 표·게이트 직후](docs/07-단계별-개발-가이드.md), [04 경계 예시](docs/04-요구사항-및-시스템설계.md), [10 UI 체크](docs/10-UI-HUD-디자인-및-구현-가이드.md), [02 §4](docs/02-준비사항-체크리스트.md)에 이어집니다. **PDF·인쇄 한 장**은 [31](docs/31-프레임워크-컨텐츠-UI-구분-및-구현.md).

#### 권장 폴더 구조

```
Assets/
├── _Framework/           ← 기반 스크립트만
│   ├── Runtime/          GameSession.cs, SceneFlow.cs, GameEvents.cs
│   └── Combat/           IDamageable.cs 등
├── _Content/
│   ├── Data/
│   ├── Prefabs/Enemies/, Prefabs/Stages/, Augments/
└── _UI/
    ├── Prefabs/, Scripts/
```

**팀 규칙**: `_Framework`는 **규칙 틀**만 — **숫자·이름·스테이지 식별**은 `_Content`의 SO에 둡니다.

#### 기반 (`_Framework/`) 체크리스트

| # | 구현 항목 | 역할 | 완료 조건 |
|:--:|-----------|------|-----------|
| 1 | **GameSession** / RunState | 런의 currentHp, maxHp, currency, 증강 ID | 씬 전환 후 유지(DontDestroyOnLoad 등) |
| 2 | **SceneFlow** | `LoadStage(string id)`, `LoadTitle()`, `ReloadRun()` | stageId만 받고 씬 매핑은 컨텐츠 테이블 가능 |
| 3 | **GameEvents** | HP·재화·증강·클리어·게임오버 | **UI는 여기만 구독** |
| 4 | **IDamageable** | `TakeDamage(int)` | 플레이어·적·보스 공통 |
| 5 | **사망·클리어 게이트** | HP 0 → GameOver, 보스 사망 → StageCleared | 규칙은 기반, **보스 여부는 컨텐츠 프리팹** |

**기반 완료**: 위가 동작하고 **UI 스크립트에 게임 수치 상수 없음**.

#### `GameEvents` 샘플 (이름은 팀 합의로 변경 가능)

```csharp
// Assets/_Framework/Runtime/GameEvents.cs
using System;
public static class GameEvents
{
    public static event Action<int, int> PlayerHpChanged;
    public static event Action<int> CurrencyChanged;
    public static event Action GameOver;
    public static event Action StageCleared;

    public static void RaisePlayerHp(int c, int m) => PlayerHpChanged?.Invoke(c, m);
    public static void RaiseCurrency(int c) => CurrencyChanged?.Invoke(c);
    public static void RaiseGameOver() => GameOver?.Invoke();
    public static void RaiseStageCleared() => StageCleared?.Invoke();
}
```

HP가 바뀔 때마다 **기반**에서만 `RaisePlayerHp` 호출 → UI는 구독만.

#### 컨텐츠 (`_Content/`) 체크리스트

| # | 항목 | 비고 |
|:--:|------|------|
| 1 | PlayerStatsData / EnemyStatsData / BossData 등 SO | [24](docs/24-기획-데이터-코드-연결-템플릿.md) 수치 |
| 2 | 스테이지 씬 + LevelSettingsData | stageId, 배경, 스폰 |
| 3 | 적·보스 프리팹 + SO | 동작은 기반 인터페이스 |
| 4 | AugmentData 목록 | 3중1택 풀 |
| 5 | 맵·장애물 | [28](docs/28-기획서-맵구성-에셋-구현-연동.md) |

**컨텐츠 완료**: **새 스테이지 = 씬 + SO만** 추가해도 기반 수정 최소.

#### UI (`_UI/`) 체크리스트

| # | 항목 | 완료 조건 |
|:--:|------|-----------|
| 1 | HUD | PlayerHpChanged 구독 → 바/텍스트만 |
| 2 | 재화 | CurrencyChanged 구독 |
| 3 | 게임 오버 | GameOver 구독 → 재시작은 SceneFlow만 호출 |
| 4 | 증강 선택 | GameSession + OnAugmentAdded — [25](docs/25-참조게임-UI-프리팹-디자인-레이아웃.md) |
| 5 | 일시정지·설정 | timeScale·볼륨만 |

**UI 금지**: `currentHp -= damage`, `if (stage == "4-3")` 같은 판정·수식.

#### 구현 순서 (폴더 기준)

| 순서 | 폴더 | 이유 |
|:--:|------|------|
| 1 | 기반 `_Framework/` | 없으면 UI·컨텐츠가 뒤엉킴 |
| 2 | 컨텐츠(최소 1스테이지·1적) | 기반에 붙일 데이터 |

#### 경계 위반 → 고치는 법

| 잘못된 예 | 올바른 분리 |
|-----------|-------------|

## Unity 씬 · 게임 흐름 · 씬 전환 · 프리팹

**폴더**: 전환·런 상태 → **`_Framework/`**, 스테이지·맵·적 → **`_Content/`**, 메뉴·HUD → **`_UI/`** — [#folder-layers](#folder-layers). **분산 정본**: [04 §2.4~2.5](docs/04-요구사항-및-시스템설계.md), [02 §4.2](docs/02-준비사항-체크리스트.md), [07 Phase 0](docs/07-단계별-개발-가이드.md), [10 Phase 4](docs/10-UI-HUD-디자인-및-구현-가이드.md). **인쇄**: 이 절(`#scene-flow`) — [32](docs/32-Unity-씬-게임흐름-씬전환-가이드.md)는 정본 위치 표.

### 씬 관점 전체 흐름

```
[랜딩/부트] ──(자동)──► [메인메뉴]
                            │ [시작]
                            ▼
              [게임 플레이] ◄────┐  재시작 / 다음 스테이지
                    │            │
         클리어/입구 │   사망    │
                    ▼            │
            (선) [증강 선택]     │
                    └────────────┤
              [게임오버] ──► [메인메뉴]
              (별 씬 또는 Game 위 패널)
```

- **최소**: 메뉴 → **Game** → 사망 시 패널 → 재시작 시 `Game` 다시 로드.
- **스테이지 다수**: `Stage_1_1` → … **씬 전환** 또는 맵만 교체 — [#map-asset-pipeline](#map-asset-pipeline) · [28](docs/28-기획서-맵구성-에셋-구현-연동.md).

### 권장 씬 목록

| 씬(예) | 필수 | 역할 |
|--------|:----:|------|
| **Landing** / Boot | 선택 | 로고 후 다음 씬 자동 로드 — Build **0번** 후보 |
| **MainMenu** | 권장 | 시작·설정·종료 |
| **Game** | 필수 | 메인 플레이(HUD는 보통 동일 씬) |
| **Stage_*** | 선택 | 스테이지별 씬 분리 시 |
| **AugmentSelect** | 선택 | 3중1택 전용 씬 — 없으면 Game 내 UI |
| **GameOver** | 선택 | 전용 씬 — **없으면 Game 씬 패널 권장** |

씬 이름은 팀이 **한 번 정해** [02](docs/02-준비사항-체크리스트.md)와 `LoadScene("이름")`에 **동일**하게 씁니다.

### Build Settings (등록 순서)

**File → Build Settings** — 위에서부터 **플레이 순서**.

| Index | 씬 |
|:-----:|-----|
| 0 | `Landing` 또는 `MainMenu` — **빌드 첫 화면** |
| 1 | `MainMenu` (Landing 쓸 때) |
| 2~ | `Game`, `Stage_1_1`, … |
| (선) | `GameOver` — 별 씬일 때만 |

- **Play 버튼**: **현재 연 씬**이 실행됨. 랜딩부터 보려면 Landing 연 상태로 Play.

### 씬 전환 — 누가 (`_Framework/`)

| 동작 | 구현 |
|------|------|
| 메뉴 → 게임 | `SceneFlow` 등에서 `SceneManager.LoadScene(..., Single)` |
| 재시작 / 메뉴 복귀 | 동일 — **UI 버튼은 `_UI/`에서 기반 public 메서드만** 호출 |
| 다음 스테이지 | `stageId` → 씬 이름은 **`_Content/` SO**로 매핑 후 LoadScene |

**Single**: 메뉴↔게임에 주로 사용. **DontDestroyOnLoad**: `GameSession`·`SceneFlow`·BGM 등 **최소만**.

### 씬별로 두면 좋은 것 (프리팹·세 폴더)

| 씬 | `_Framework/` | `_Content/` | `_UI/` |
|----|----------------|-------------|--------|
| Landing | (선) 부트 | — | (선) 로고 |
| MainMenu | 싱글톤 두기 좋음 | — | Canvas·시작 버튼 |
| Game / Stage_* | (이전 씬 유지 가능) | 맵·적·Player 스폰 | HUD·게임오버 패널 |
| GameOver | — | — | 다시하기·메뉴 |

**Player**: 씬 배치 또는 `SpawnPoint`에서 `Instantiate(Player.prefab)`. HP 표시는 [10](docs/10-UI-HUD-디자인-및-구현-가이드.md) — **기반 이벤트만** 구독.

### Phase별 — 씬·전환 할 일

| Phase | 할 일 |
|:-----:|------|
| **0** | Landing(선)·MainMenu·Game **빈 씬** → Build 등록 → **메뉴→Game** 전환만 검증 |
| **1** | Game에 Player·바닥 |
| **2** | 맵·장애물 |
| **3** | 적·전투·사망 시 기반에서 게임오버 이벤트 |
| **4** | 재시작·메뉴 버튼 → **SceneFlow** |
| **5** | 빌드 후 **index 0** 확인 ([19](docs/19-분야별-분리-통합-및-제품출시-가이드.md)) |

### SceneFlow 최소 예 (`_Framework/`)

```csharp
using UnityEngine;
using UnityEngine.SceneManagement;

public class SceneFlow : MonoBehaviour
{
    public void GoToMainMenu() => SceneManager.LoadScene("MainMenu");
    public void GoToGame() => SceneManager.LoadScene("Game");
    public void ReloadCurrentStage() =>
        SceneManager.LoadScene(SceneManager.GetActiveScene().name);
}
```

씬 이름 문자열은 **상수** 또는 **`_Content/` SO**(stageId → 씬명)로 빼면 F-12·[04](docs/04-요구사항-및-시스템설계.md)와 맞습니다.

**[03 게임기획서 초안](docs/03-게임기획서-초안.md)을 따르는 팀**은 스테이지 전환·증강 선택·장애물·상점·재화 등으로 **인터랙션이 필수**입니다 — [04 F-12·F-15](docs/04-요구사항-및-시스템설계.md). **01만으로 최소 플랫포머**만 만들 팀은 일부를 생략할 수 있으나, **아래 순서**를 알아 두면 UI·사운드와 꼬이지 않습니다.

**압사·입구·체크포인트·구역 스폰·버튼·전투 판정** 등은 전부 **“무언가에 닿거나 조건이 맞을 때 뭔가 실행”**입니다.

## 전체 컨텐츠 검증 — 산나비? 초심자 전제?

| 질문 | 검증 결과 |
|------|-----------|
| **이 문서대로 하면 산나비 같은 게임이 되나?** | **아니요(동일 작품 불가).** 산나비는 **사슬(훅) 스윙·그래플링**이 **핵심 조작**인 상업 인디작입니다. 본 가이드는 **이동·점프·근접(또는 단순) 공격** 중심의 **2D 플랫포머 + 로그라이크(증강·스테이지)** 입니다. 기획서·01에 **산나비는 비주얼·분위기 레퍼런스**로만 적혀 있으며, **훅 이동·내러티브·연출 규모**는 문서 범위에 **포함되지 않습니다**. |
| **그럼 무엇까지가 문서 범위인가?** | [03 게임기획서 초안](docs/03-게임기획서-초안.md)에 가까운 **한 줄 루프**: 스테이지 진행 → **3중1택 증강** → 전투·맵·UI → (선)보스 → **PC 빌드**. [01](docs/01-요구분석.md) 필수 기능(이동·점프·적·HP·게임오버 등)이 **실제로 돌아가는 수준**까지가 목표입니다. |
| **게임 개발 초심자가 따라 써도 되나?** | **전제: 문서 순서·Phase 완료 조건을 지킬 것.** 그 전제에서 **교육·팀 과제·첫 포폴용 프로토타입**에 맞게 설계되어 있습니다. **혼자·첫 유니티**도 가능하나, 산나비급 그래픽·스토리·폴리싱은 **별도 학습·기간**이 필요합니다. |
| **산나비에 더 가깝게 가려면?** | 훅 물리·레벨 디자인은 **추가 모듈(문서 미포함)**. 먼저 본 가이드로 **플랫폼+전투 루프**를 완주한 뒤, 훅 전용 튜토리얼·프로토타입을 더하는 방식을 권장합니다. |

**전제 (아래 표 공통)**  
- **순개발 투입시간**: 문서 읽기·Unity·C#·디버깅·Play 검증에 쓰는 시간만 합산 (**수업·회의·대기·아트 제작은 미포함**).

- **통상 하루 1~2시간**만 넣을 수 있다면 → **주 5~10시간** 정도(평일만 1h×5 ~ 2h×5)로 보면 아래 표의 **6~10시간/주**에 가깝습니다. **주말만**이면 **주 2~4시간** 행을 보세요.

## 웹 사이트 (GitHub Pages) — 이 저장소가 그대로 올라감

`docs/`에 **Docsify** (`index.html` + 마크다운)가 있으며, **`main`(또는 `master`)에 푸시할 때마다** [.github/workflows/deploy-github-pages.yml](.github/workflows/deploy-github-pages.yml) 이 배포합니다.

### 최초 1회 (저장소 관리자)

1. GitHub 저장소 → **Settings** → **Pages**
2. **Build and deployment** → **Source**: **GitHub Actions** (브랜치 `/docs`가 아님)
3. **Actions** 탭에서 **Deploy GitHub Pages** 워크플로를 한 번 실행(또는 `main`에 푸시)

### 주소

`https://<계정>.github.io/<저장소이름>/`  
예: `https://joygr.github.io/roguelike-lect/`

### 로컬 미리보기

```bash
cd docs && npx --yes serve .
```

### 테마 변경 시 코드 색

`docs/index.html`에서 Docsify 테마 CSS(`vue.css` → `dark.css` 등)만 바꿔도, **`docs/assets/github-pages-code.css`** 가 블록 코드·구문 강조·인라인 코드 대비를 고정해 두어 소스 색이 깨지지 않게 합니다.
