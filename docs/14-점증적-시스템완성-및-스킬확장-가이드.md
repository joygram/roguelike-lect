# 점증적 시스템 완성 및 스킬 확장

**시스템이 점증적으로 완성**되도록 하는 원칙과,  
**스킬·아이템·몬스터** 등을 **하나씩 확장**할 수 있게 하기 위한 **내부 구조**를 정리한 문서입니다.

# Part 1. 점증적 시스템 완성

## 1.1 원칙

- **점증적 완성**: 각 Phase(또는 스프린트)가 끝날 때 **동작하는 조각**이 하나씩 쌓입니다.  
  “반쯤 만든 기능”을 여러 개 두지 않고, **한 번에 하나의 흐름**이 끝까지 동작하도록 만듭니다.
- **완성된 조각**:  
  - Phase 1 끝 → “이동 + 카메라”만 있어도 **한 조각 완성**.  
  - Phase 2 끝 → “맵 생성 + 그 위에서 이동”이 **한 조각 완성**.  
  - Phase 3 끝 → “몬스터 스폰 + 전투 + 사망”이 **한 조각 완성**.  
  이렇게 **이전 단계는 그대로 두고**, 위에만 새 동작을 붙입니다.
- **미완성 기능 배제**: “나중에 쓸 거니까” 하고 코드만 넣고 데이터·연결이 없으면, 그 Phase에서는 **비활성화**하거나 **다음 Phase에서 추가**합니다.  
  매 Phase마다 **실행 가능한 상태**를 유지합니다.

---

## 1.2 Phase별 “완성된 조각” 요약

| Phase | 완성되는 조각 | 확장 포인트 |
|-------|----------------|-------------|
| 0 | 프로젝트·씬 실행 | — |
| 1 | 입력 → 이동 → 카메라 | (이동만 있음) |
| 2 | 맵 생성 → 그 위에서 이동·충돌 | 맵 크기·방 개수는 데이터로 확장 |
| 3 | 몬스터 스폰 → AI → 전투 → 사망/피격 | 몬스터 종류는 데이터로 확장 |
| 4 | HP 표시 → 게임 오버 → 재시작 | — |
| 5 | 사운드·이펙트 연결 | 효과음·이펙트 종류는 데이터로 확장 |

→ **점증적 완성** = 위 표의 “완성되는 조각”이 순서대로 **한 번에 하나씩** 확정되는 것입니다.

---

## 1.3 확장 가능한 부분 정리

아래는 “**내부 구조를 한 번 완성해 두면**, 이후에는 **데이터(에셋)만 추가**해서 하나씩 확장할 수 있는 부분입니다.

| 대상 | 내부 구조 (한 번 완성할 것) | 확장 방법 (하나씩 추가) |
|------|-----------------------------|-------------------------|
| **스킬** | SkillData(ScriptableObject) + 스킬 목록 + “목록에서 하나 골라 실행” 로직 | 새 SkillData 에셋 생성 → 목록에 추가 |
| **몬스터** | EnemyStatsData + 스폰 시 stats만 교체 | 새 EnemyStatsData 에셋 생성 → 스폰 테이블에 추가 |
| **아이템** | ItemData + 획득 시 “종류별 효과” 분기 | 새 ItemData 에셋 생성 → 드롭 테이블에 추가 |
| **맵** | DungeonSettingsData + 그리드 생성 로직 | 설정값·시드만 바꿔서 맵 다양화 |

이 문서에서는 **스킬**을 예로 들어, 내부 구조가 어떻게 완성되고, 그 다음에는 어떻게 “하나씩 자연스럽게” 확장하는지 구체적으로 적습니다.

---

# Part 2. 스킬 확장 — 내부 구조와 자연스러운 확장

## 2.1 목표

- **스킬을 하나씩 확장**할 수 있어야 함.  
- **내부 구조가 완성**되면, 새 스킬 추가는 **코드 수정 없이**(또는 최소한으로) **에셋만 추가**하면 되도록 함.  
- “스킬 1개 동작 → 스킬 2개 동작 → 스킬 3개 …”가 **같은 코드**로 처리되도록 함.

---

## 2.2 내부 구조 (한 번 완성할 것)

스킬 쪽 **내부 구조**는 아래 세 가지가 갖춰지면 “완성”으로 봅니다.

1. **스킬 데이터 형식**  
   - 스킬 하나당 필요한 정보를 담는 **ScriptableObject** (예: `SkillData`).  
   - 데미지, 쿨타임, 이펙트 프리팹, 사운드, (선택) 스킬 ID·이름 등.
2. **스킬 목록**  
   - 플레이어(또는 스킬 매니저)가 “가지고 있는 스킬들”을 **목록(배열/리스트)** 로 보관.  
   - `SkillData[]` 또는 `List<SkillData>`.
3. **실행 로직**  
   - “입력/버튼 → 목록에서 N번째 스킬 실행” 또는 “입력 → 해당 스킬 데이터로 공격 처리”.  
   - 모든 스킬이 **같은 실행 경로**(한 함수)를 타고, **데이터만 다르게** 동작하도록 함.

이 구조가 들어가면, **새 스킬 = 새 SkillData 에셋 + 목록에 한 개 추가**만 하면 되어 **자연스럽게 확장**됩니다.

---

## 2.3 SkillData (스킬 데이터 형식)

스킬 하나의 “설정”만 담는 에셋입니다. 게임로직은 이 **형식만** 알고, 실제 값은 모두 에셋에서 읽습니다.

```csharp
// SkillData.cs — 스킬 하나의 데이터 (데이터 담당이 에셋 생성)
[CreateAssetMenu(fileName = "Skill", menuName = "Roguelike/Skill Data")]
public class SkillData : ScriptableObject
{
    public string skillName = "기본 공격";
    public int damage = 10;
    public float cooldown = 0.5f;
    public float range = 1.5f;           // (선택) 사거리
    public GameObject effectPrefab;       // (선택) 발동 이펙트
    public AudioClip soundClip;           // (선택) 발동 사운드
}
```

- **확장 시**: 새 스킬을 넣고 싶으면 **새 SkillData 에셋**을 만들고, damage/cooldown/effect/sound만 바꾸면 됩니다. **코드는 수정하지 않습니다.**

---

## 2.4 스킬 목록 + 실행 로직 (게임로직 쪽)

플레이어(또는 전투 담당 스크립트)가 **스킬 목록**을 가지고, **인덱스 또는 키에 따라** 그중 하나를 실행합니다.

```csharp
// PlayerCombat.cs (요지만)
public class PlayerCombat : MonoBehaviour
{
    public SkillData[] skills;        // 스킬 목록 (Inspector에서 에셋 할당, 1개부터 N개까지)
    public KeyCode[] skillKeys;       // skills[0] → skillKeys[0], 순서대로 매칭 (선택)
    float[] nextUseTime;             // 스킬별 쿨타임

    void Start()
    {
        if (skills == null) return;
        nextUseTime = new float[skills.Length];
    }

    void Update()
    {
        for (int i = 0; i < skills.Length; i++)
        {
            if (skills[i] == null) continue;
            KeyCode key = (skillKeys != null && i < skillKeys.Length) ? skillKeys[i] : KeyCode.None;
            if (key != KeyCode.None && Input.GetKeyDown(key))
                TryUseSkill(i);
        }
    }

    void TryUseSkill(int index)
    {
        if (index < 0 || index >= skills.Length || skills[index] == null) return;
        SkillData s = skills[index];
        if (Time.time < nextUseTime[index]) return;

        nextUseTime[index] = Time.time + s.cooldown;
        DoSkill(s);   // 실제 데미지·이펙트·사운드는 데이터(s)만 사용
    }

    void DoSkill(SkillData s)
    {
        // 기존 공격 로직: s.damage, s.range 사용
        // 이펙트: s.effectPrefab, 사운드: s.soundClip
    }
}
```

- **스킬 1개**: `skills`에 1개만 넣으면 1개만 동작.  
- **스킬 2개, 3개 …**: `skills`에 에셋만 더 넣고, `skillKeys`만 맞춰 주면 됩니다.  
→ **내부 구조가 완성되면**, 스킬 추가는 **에셋 + 배열 길이**만 바꾸는 **자연스러운 확장**이 됩니다.

---

## 2.5 스킬 확장 순서 (실무)

| 단계 | 할 일 | 결과 |
|------|--------|------|
| 1 | `SkillData` 클래스 정의, **스킬 1개** 에셋 생성 (예: 기본 공격) | 한 개 스킬만 동작 |
| 2 | `skills` 배열(또는 리스트) + `TryUseSkill(index)`, `DoSkill(SkillData)` 구현 | “목록의 N번째 스킬 실행” 구조 완성 |
| 3 | **스킬 2개** 에셋 만들어서 `skills`에 2개 할당, 키 2개 매핑 | 스킬 2개 동작 (코드 수정 없음) |
| 4 | 이후 스킬 3, 4, … 도 **에셋 추가 + 배열에 할당**만 반복 | 하나씩 확장 |

→ **내부 구조(데이터 형식 + 목록 + 실행 로직)** 가 먼저 완성되면, 그 다음부터는 **스킬을 하나씩 확장**할 때 코드를 거의 건드리지 않아도 됩니다.

---

## 2.6 (선택) 키 매핑을 데이터로

“이 키는 이 스킬”을 코드가 아니라 **데이터**로 두고 싶다면, 스킬에 키를 넣을 수 있습니다.

```csharp
// SkillData에 추가
public KeyCode key = KeyCode.X;
```

- `PlayerCombat`에서는 `skills` 배열을 돌면서 `skills[i].key`로 입력을 받으면 됩니다.  
- 새 스킬 추가 시 **에셋에서 키만 지정**하면 되므로, 확장이 더 자연스럽습니다.

# Part 3. 다른 요소의 “하나씩 확장” (몬스터·아이템)

같은 생각으로 **몬스터**·**아이템**도 “내부 구조 완성 → 데이터만 추가”로 확장할 수 있습니다.

## 3.1 몬스터

- **내부 구조**: `EnemyStatsData` + 스폰 시 “이 방에는 EnemyStatsData 목록 중 하나(또는 N개)”를 골라서 인스턴스에 할당.  
- **확장**: 새 몬스터 = **새 EnemyStatsData 에셋** + 스폰 테이블/목록에 추가.  
- 이미 09 문서의 “1종 프리팹 + 데이터만 변경”과 동일한 패턴입니다.

## 3.2 아이템

- **내부 구조**: `ItemData` (ScriptableObject) + “획득 시 itemType/effect 값에 따라 분기”하는 한 곳.  
- **확장**: 새 아이템 = **새 ItemData 에셋** + (필요 시) 드롭 테이블에 추가.  
- 효과 종류(회복, 버프 등)는 처음에 1~2가지만 구현해 두고, 나중에 `ItemData`에 필드만 추가해도 “하나씩 확장”이 가능합니다.

# Part 4. 한눈에 보기

| 항목 | 내용 |
|------|------|
| **점증적 완성** | Phase마다 **동작하는 조각** 하나씩 완성. 반쯤 만든 기능을 여러 개 두지 않음. |
| **스킬 하나씩 확장** | `SkillData`(에셋) + `skills`(목록) + `TryUseSkill(index)` / `DoSkill(SkillData)` 로직 → 새 스킬 = 새 에셋 + 목록에 추가. |
| **내부 구조 먼저** | 데이터 형식(SkillData) + 목록 + “목록 기준 실행”이 완성되면, 이후 확장은 **데이터만** 추가. |
| **자연스러운 확장** | 코드는 그대로 두고, 에셋·배열 길이·키 매핑만 바꿔서 스킬 2, 3, 4… 가 동작하도록 함. |
| **몬스터·아이템** | 같은 패턴: Stats/Item 데이터 + “목록/테이블에서 하나 골라 사용” → 새 종류 = 새 에셋 + 테이블 추가. |

이 순서대로 하면 **시스템이 점증적으로 완성**되고, **스킬 등은 내부 구조가 갖춰진 뒤 하나씩 자연스럽게 확장**할 수 있습니다.
