# 03. 맵 — 기획

> **시스템**: 환경·맵 | **담당**: P1 (맵 로직 설계) · D2 (타일 구성 합의)  
> 업무 분장 전체 기준 → [팀 역할 배분 문서](팀개발-역할배분-개발단계.md)

이동과 점프만 있으면 플레이어가 달릴 공간이 없다. 맵은 게임 플레이가 실제로 일어나는 무대다. 이 단계에서는 어떤 방식으로 맵을 구성할지, 어떤 종류의 지형이 필요한지를 결정한다.

```
                                      ↑ 기획 (지금)
```

## 맵 구성 방식 결정

맵을 만드는 방법은 크게 두 가지다.

### 고정 스테이지
에디터에서 직접 타일을 그려서 맵을 만든다. 플레이어가 볼 때마다 같은 맵이 나온다.

- **장점**: 구현이 빠르다. 맵 디자인을 세밀하게 컨트롤할 수 있다
- **단점**: 반복 플레이 시 새로운 느낌이 없다

### 랜덤 생성 (절차적 생성)
코드로 맵을 자동으로 생성한다. 매판 다른 맵이 나온다.

- **장점**: 반복해도 새로운 경험
- **단점**: 구현 난이도가 높다. 막히거나 진행 불가능한 맵이 생길 수 있다

**이 강좌에서는 고정 스테이지로 시작한다.** 기본을 완성한 뒤 원하면 랜덤 생성으로 확장하면 된다.

---

## 필요한 지형 종류

어떤 지형이 있어야 게임이 재미있을지 생각해보자.

| 지형 종류 | 역할 | 물리 설정 |
|----------|------|----------|
| **바닥** | 플레이어·적이 밟는 기본 지면 | 콜라이더 있음 |
| **플랫폼** | 높이 있는 발판. 아래에서 점프해서 올라설 수 있음 | 단방향 콜라이더 (위에서만 착지) |
| **벽** | 좌우 이동을 막는 수직 지형 | 콜라이더 있음 |
| **배경 장식** | 충돌 없이 뒤에 보이는 장식 | 콜라이더 없음 |

단방향 플랫폼은 "아래에서는 통과되고, 위에서는 막히는" 발판이다. Unity의 `PlatformEffector2D` 컴포넌트로 구현한다.

---

## 스테이지 설계

한 스테이지가 얼마나 넓어야 할지를 먼저 정한다.

**권장 시작 크기:**
- 가로: 화면 3~5개 분량 (약 60~100 units)
- 세로: 화면 1~2개 분량 (약 20~40 units)

너무 크면 맵을 채우는 작업이 오래 걸린다. 처음엔 작게 만들고, 게임이 완성되면 크기를 늘리자.

아래는 기본 스테이지 스케치 예시다. 종이에 먼저 그려보고 에디터에서 구현하면 훨씬 빠르다.

```
     ↑ 스폰                      [목표?]
```

## 플레이어 스폰 위치

플레이어가 씬 시작 시 어디에서 나타날지를 정한다. 보통 왼쪽 하단이다. 씬에 빈 오브젝트 `SpawnPoint`를 원하는 위치에 배치해서 기준점으로 쓴다.

## 에셋 담당에게 필요한 것

| 항목 | 설명 |
|------|------|
| 바닥 타일 | 32×32px, 반복 배치할 수 있는 패턴 |
| 벽 타일 | 32×32px |
| 플랫폼 타일 | 32×32px, 위에서만 밟는 발판용 |
| 배경 타일 | 충돌 없는 장식용 (선택) |
