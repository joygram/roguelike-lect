# 기획서 → 개발용 완성 단계

**원래(이미 작성한) 게임기획서**를 놓고, **누락된 부분만 찾아서 설계 쪽을 살짝만 채우면**  
**개발에 필요한 모든 내용**이 갖춰진 기획서로 완성할 수 있도록 하는 **단계 문서**입니다.

- **체크리스트가 아님**: “항목 체크”가 아니라 **기존 기획서를 분석**해 **어디가 비었는지**를 먼저 파악합니다.
- **설계는 살짝만**: 이미 잘 적혀 있는 항목은 고치지 않고, **없거나 부족한 부분만** 최소한으로 채웁니다.
- **중간에 설명**: 각 단계·섹션마다 “왜 이 내용이 개발에 필요한지”, “어디에서 쓰이는지”를 짧게 설명합니다.

기획서를 **처음부터 새로 쓰는** 경우는 [17-기획-단계별-실제](17-기획-단계별-실제-가이드.md)를,  
**보강·개선·추가**를 항목별로 채우는 경우는 [21-기획서-보강-개선-추가](21-기획서-보강-개선-추가-가이드.md)를 사용하세요.

# Part 0. “개발에 필요한 모든 내용”이란

기획서가 **개발 단계에서 그대로 쓰이려면** 아래 항목들이 **빠지지 않고** 있어야 합니다.  
이미 있는지 여부는 Part 1에서 점검하고, 없는 부분만 Part 2에서 채웁니다.

| 구분 | 개발에 필요한 항목 | 개발에서 쓰이는 곳 (참고) |
|------|-------------------|---------------------------|
| **개요** | 게임명, 장르, 대상 유저, 플랫폼, 엔진, 개발 기간, (선택) 레퍼런스 | 팀 목표 공유, 02·05 에셋·환경 |
| **핵심 플레이** | 플레이어 목표, 조작(키/버튼), 승리 조건, 패배 조건, (선택) 한 판 플레이 타임 | 04 요구사항, 07 Phase 목표 |
| **스테이지·맵** | 고정/소수 스테이지, 플랫폼·바닥·타일 종류 | 04 F-04, 08 스테이지/레벨 설정 테이블 |
| **적** | 종류 수, 종류별 한 줄 특징·행동 | 04 F-05~F-08, 08 적 테이블 |
| **플레이어·전투** | HP 유무, 이동·점프·공격, 스킬 유무·개수, 적 처치 후 드롭 유무 | 04 F-01~F-03, 스킬·데미지 설계 |
| **아이템·강화** | 유무, 있으면 종류별 한 줄 (없으면 “없음” 명시) | 05 에셋, 08 아이템 테이블 |
| **데이터(수치)** | 플레이어/적/스킬/스테이지 수치·규칙 테이블 | 08 데이터 테이블, Phase 2·3 |
| **에셋·기술** | 엔진·플랫폼, 필요 에셋 종류(캐릭터·맵·UI·사운드) | 02·05, 빌드·테스트 환경 |
| **범위·일정** | 팀 규모·기간에 맞는 범위인지 한 줄 | 07 Phase 구간, 범위 조정 |

→ **시스템 목록·요구사항 ID**는 기획서 본문이 채워진 뒤 04 문서에서 정리하면 됩니다.  
  이 문서에서는 **기획서 본문에 “무엇이 있어야 개발에 쓸 수 있는지”**만 채우는 데 집중합니다.

# Part 1. 1단계 — 현재 기획서 분석

**목적**: 원래 기획서를 한 번씩 대조해서, **어떤 항목이 있고 / 없고 / 부족한지** 표로 정리합니다.  
이 표가 있으면 Part 2에서 “없음·부족”인 항목만 골라서 채우면 됩니다.

## 1.1 분석 표 사용 방법

- 팀 기획서(또는 첨부한 기획서)를 열어 두고, 아래 표의 **각 행**을 하나씩 확인합니다.
- **있음**: 기획서에 해당 내용이 명확히 적혀 있음.  
- **부족**: 있긴 한데 모호하거나, 개발에 쓰기엔 구체성이 부족함.  
- **없음**: 해당 내용이 없음.
- **비고** 칸에는 “2장에만 있음”, “수치는 없고 설명만 있음”처럼 **어디에 어떻게 있는지** 짧게 적어 두면, 나중에 채울 때 참고하기 좋습니다.

## 1.2 기획서 항목별 분석 표

**아래 표를 복사해, 우리 기획서 기준으로 “있음 / 부족 / 없음”과 비고를 채우세요.**

| 대분류 | 항목 | 있음 / 부족 / 없음 | 비고 (위치·상태) |
|--------|------|:------------------:|------------------|
| **개요** | 게임명 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 장르(로그라이크 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 대상 유저(연령·경험) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 플랫폼(PC 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 엔진·버전(Unity 6 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 개발 기간(주·차시) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | (선택) 참고 레퍼런스 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **핵심 플레이** | 플레이어 목표(한 문장) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 조작(이동·공격·스킬 키) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 승리 조건 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 패배 조건·게임 오버 후 재시작 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | (선택) 한 판 플레이 타임 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **스테이지·맵** | 스테이지 구성(고정/소수 레벨) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 플랫폼·방 크기·타일(바닥·벽 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **적** | 종류 수(최소 1~2종) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 종류별 한 줄 특징·행동 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **전투·플레이어** | 플레이어 HP·이동·점프·공격 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 공격 방식(접촉/버튼 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 스킬 유무·개수 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 적 처치 후 드롭 유무 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **아이템·강화** | 유무 명시, 있으면 종류별 한 줄 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **데이터** | 플레이어 수치(HP·속도·점프력·공격력 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 적 수치(HP·속도·공격력 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 스킬 정의(데미지·쿨·사거리 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 스테이지/맵 설정(맵 크기 등) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **에셋·기술** | 엔진·플랫폼 명시 | ☐ 있음 ☐ 부족 ☐ 없음 | |
| | 필요 에셋 종류(캐릭터·맵·UI·사운드) | ☐ 있음 ☐ 부족 ☐ 없음 | |
| **범위** | 팀 규모·기간에 맞는 범위 한 줄 | ☐ 있음 ☐ 부족 ☐ 없음 | |

## 1.3 분석 결과 정리

- **“없음”·“부족”**으로 표시한 행만 Part 2에서 해당 섹션으로 가서 **누락된 부분만** 채우면 됩니다.
- **“있음”**인 항목은 **수정하지 않아도** 개발에 쓸 수 있다고 보면 됩니다.  
  → 설계를 “살짝만” 채우는 것이므로, 이미 있는 내용은 유지하는 것이 원칙입니다.

# Part 2. 2단계 — 누락된 부분만 채우기

Part 1에서 **없음·부족**으로 표시한 **대분류**만 골라서, 아래 해당 섹션으로 이동합니다.  
각 섹션에는 **왜 개발에 필요한지** 짧은 설명과, **누락 시 여기만 채우면 되는** 최소한의 작성 칸이 있습니다.  
이미 기획서에 잘 적혀 있으면 **건너뛰어도** 됩니다.

---

## 2.1 개요

**왜 필요한지**  
개요가 있으면 팀이 “무슨 게임을, 누구에게, 어떤 도구로, 언제까지” 만드는지 공유할 수 있고, 02 준비사항·05 에셋 목록에서 환경·리소스 범위를 정할 수 있습니다.

**개발에서 쓰임**  
02(Unity 버전·폴더), 05(에셋 종류·개수), 07(Phase 기간 가정).

**누락 시 여기만 채우기** (한 줄씩만 써도 됨)

| 항목 | 우리 기획서에 넣을 내용 (최소) |
|------|--------------------------------|
| 게임명 | |
| 장르 | (예: 로그라이크, 탑다운 액션) |
| 대상 유저 | (예: 초등학생, 입문자) |
| 플랫폼 | (예: PC Windows/Mac) |
| 엔진·버전 | (예: Unity 6.3 LTS) |
| 개발 기간 | (예: 8주, 12차시) |
| (선택) 레퍼런스 | (게임명 또는 URL) |

---

## 2.2 핵심 플레이 (목표·조작·승리·패배)

**왜 필요한지**  
“플레이어가 뭘 하고, 어떻게 조작하고, 언제 이기고 지는지”가 문장으로 있어야 04 요구사항과 07 Phase별 목표를 정할 수 있습니다. 흐리면 “재미의 중심”이 팀마다 다르게 해석됩니다.

**개발에서 쓰임**  
04(F-01 조작·이동, F-02 공격, F-09 게임 오버 등), 07 Phase 1~4 목표 문장.

**누락 시 여기만 채우기**

| 항목 | 우리 기획서에 넣을 문장 (한 줄) |
|------|--------------------------------|
| 플레이어 목표 | “플레이어는 ○○를 해서 △△한다.” |
| 조작 | 이동: __ / 공격: __ / 스킬: __ (키 이름) |
| 승리 조건 | (또는 “승리 조건 없음, 생존 목표”) |
| 패배 조건 | (예: HP 0 → 게임 오버, 재시작 가능) |
| (선택) 한 판 플레이 타임 | (예: 5~15분) |

---

## 2.3 스테이지·맵(플랫폼)

**왜 필요한지**  
스테이지가 고정인지, 플랫폼·바닥이 어떻게 구성되는지가 있어야 08 스테이지/레벨 설정 테이블과 Phase 2 맵 배치를 설계할 수 있습니다. 너무 크거나 복잡하면 기간 안에 완성하기 어렵기 때문에 “우리 규모로 만들 수 있는 수준”으로만 적으면 됩니다.

**개발에서 쓰임**  
04 F-04(스테이지·플랫폼 맵), 08 스테이지/레벨 설정 테이블, 07 Phase 2.

**누락 시 여기만 채우기**

| 항목 | 우리 기획서에 넣을 내용 (한 줄~표) |
|------|------------------------------------|
| 스테이지 구성 | (고정 맵 1~3개 / 소수 레벨) |
| 플랫폼·바닥·벽 | (타일맵 또는 프리팹, 바닥·벽 타일) |
| 맵 크기 | (예: 가로 40칸, 세로 30칸) |

---

## 2.4 적

**왜 필요한지**  
적이 몇 종인지, 어떻게 움직이고 플레이어에게 어떻게 데미지를 주는지가 있어야 08 적 테이블과 Phase 3 AI·전투 구현 범위가 정해집니다. 1~2종으로 잡고, “느리지만 튼튼”처럼 한 줄만 있어도 됩니다.

**개발에서 쓰임**  
04 F-05~F-08(적 스폰·AI·전투), 08 적 수치 테이블, 07 Phase 3.

**누락 시 여기만 채우기**

| 항목 | 우리 기획서에 넣을 내용 |
|------|-------------------------|
| 종류 수 | (예: 2종) |
| 종류별 한 줄 | 이름: __ / 특징·행동: __ (예: 슬라임—느리지만 튼튼, 접촉 데미지) |

---

## 2.5 플레이어·전투

**왜 필요한지**  
플레이어 HP 유무, 공격 방식(접촉/버튼), 스킬 개수, 적 처치 후 드롭 유무가 있으면 전투 로직·스킬·데미지 수치를 08 테이블과 07 Phase 3에서 설계할 수 있습니다.

**개발에서 쓰임**  
04 F-01~F-03(이동·공격·HP), 08 플레이어·스킬 테이블, 07 Phase 3·4.

**누락 시 여기만 채우기**

| 항목 | 우리 기획서에 넣을 내용 |
|------|-------------------------|
| 플레이어 | HP 있음, 이동·공격 가능 (스킬 있으면 개수만: 예 2개) |
| 공격 방식 | (접촉 시 데미지 / 공격 버튼 데미지) |
| 스킬 | (유무, 있으면 “기본 공격 + 스킬 1~2개” 등) |
| 적 처치 후 | (드롭 있음/없음, 있으면 “코인·경험치” 등 한 줄) |

---

## 2.6 아이템·강화 (선택)

**왜 필요한지**  
아이템·강화가 있으면 “어떤 종류(회복, 강화 등)”인지 한 줄이라도 있어야 05 에셋·08 아이템 테이블 범위가 정해지고, 없으면 “1차에는 없음”이라고만 적어 두면 개발 범위가 명확해집니다.

**개발에서 쓰임**  
05 에셋 목록, 08 아이템 정의 테이블(있을 때).

**누락 시 여기만 채우기**

| 항목 | 우리 기획서에 넣을 내용 |
|------|-------------------------|
| 유무 | (있음 / 1차에는 없음) |
| 있으면 | 종류별 한 줄 (예: 빨간 포션—HP 회복, 코인—점수) |

---

## 2.7 데이터(수치·규칙)

**왜 필요한지**  
플레이어·적·스킬·스테이지의 **수치·규칙**이 테이블로 있으면 08 데이터 테이블과 Phase 2·3 구현에서 그대로 쓸 수 있습니다. 기획서에 “문장 설명만 있고 수치는 없다”면 여기서 **최소한의 수치만** 채워 넣으면 됩니다.

**개발에서 쓰임**  
08 데이터 테이블·맵 구현, 07 Phase 2(스테이지·맵), Phase 3(전투·적).

**누락 시 여기만 채우기** (숫자만 넣어도 됨)

- **플레이어**: 최대 HP __, 이동 속도 __, 기본 공격력 __, 공격 쿨(초) __  
- **적**(종류별): 이름 / HP / 이동 속도 / 공격력 (한 행씩)  
- **스킬**: 이름 / 키 / 데미지 / 쿨(초) / 사거리·범위 (한 행씩)  
- **스테이지/맵**: 맵 가로·세로(칸), 플랫폼·바닥 규칙  

→ 자세한 표 형식은 [17-기획-단계별-실제](17-기획-단계별-실제-가이드.md) Part 3, [08-데이터테이블-및-맵구현](08-데이터테이블-및-맵구현-가이드.md)를 참고하세요. 여기서는 **설계 쪽을 살짝만** 채우면 되므로, “대략 이 정도 수치를 쓴다”는 한 줄 요약만 기획서에 넣어도 됩니다.

# Part 3. 완료 조건과 다음 단계

## 3.1 이 문서까지 완료한 상태

- Part 1에서 **현재 기획서**를 분석해 “있음 / 부족 / 없음”을 표로 채웠다.
- Part 2에서 **없음·부족**인 항목만 골라, 해당 섹션의 “누락 시 여기만 채우기”를 기획서에 **반영**했다.
- 그 결과, **개발에 필요한 항목**(Part 0 표)이 기획서에 빠짐없이 들어간 상태다.

→ 이 상태를 “**기획서가 개발용으로 완성된 상태**”로 볼 수 있습니다.  
  설계·스펙은 **이미 있던 것은 유지**하고 **빈 부분만 최소한으로** 채웠기 때문에, “살짝만 채우기”에 맞게 진행된 것입니다.
