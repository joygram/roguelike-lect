# 직무별 문서·역할 및 스테이지별 연결·동작 보장

**직무별로 어떤 문서를 참고할지**, **게임로직·데이터 담당을 어떻게 나누고 연결할지**,  
**각 개발 스테이지에서 무엇을 완성하고 어떻게 동작을 보장할지**를 정리한 문서입니다.

---

# Part 2. 게임로직·데이터 담당 분리 및 연결

## 2.1 원칙

- **게임로직**: “어떤 때에 무엇을 할지”만 담당. **수치·클립·프리팹 참조**는 **데이터(에셋)** 에서 가져옵니다.
- **데이터**: **형식(클래스·필드)** 을 정하고, 그에 맞는 **에셋**을 만듭니다. 게임로직은 그 형식만 알면 됩니다.
- **연결**: 게임로직 스크립트에 `public PlayerStatsData stats;` 처럼 **데이터 에셋 참조**를 두고, Inspector에서 데이터 담당이 만든 에셋을 할당합니다. 또는 **공통 데이터 폴더**를 정해 두고, 게임로직은 “이름으로 로드”하지 않고 **참조만** 사용합니다.

이렇게 나누면 **게임로직**은 “동작 코드”만, **데이터**는 “수치·리소스”만 담당하고, **스테이지 완료 시 참조만 연결해도** 동작이 이어집니다.

---

## 2.2 인터페이스 정의 (게임로직 ↔ 데이터)

아래는 **데이터 쪽이 제공하는 형식**이고, **게임로직 쪽은 이 형식만 사용**합니다.

| 데이터 종류 | 제공 형식 (데이터 담당) | 사용처 (게임로직) |
|-------------|-------------------------|-------------------|
| 플레이어 수치 | `PlayerStatsData` (ScriptableObject): maxHp, moveSpeed, attackPower, attackCooldown | PlayerController, PlayerHealth, PlayerCombat |
| 적 수치 | `EnemyStatsData`: enemyName, maxHp, moveSpeed, damage, scoreValue, expValue | EnemyController, 스폰 로직 |
| 던전 설정 | `DungeonSettingsData`: width, height, roomCountMin/Max, roomSizeMin/Max, corridorWidth | DungeonGenerator |
| (선택) 스킬/타격 | `AttackHitData`: hitSound, hitEffectPrefab | PlayerCombat, 전투 로직 |

**연결 방법 (가장 단순)**  
- 게임로직 스크립트에 `public PlayerStatsData stats;` 등 **public 필드**를 두고,  
- 씬/프리팹에서 **데이터 담당이 만든 에셋**을 Inspector로 할당.  
- 코드에서는 `stats.maxHp`, `stats.moveSpeed` 등 **필드만 읽어서 사용**.  
- 데이터 담당은 **같은 클래스·필드명**을 유지한 채 에셋 값만 수정하면, 게임로직 수정 없이 밸런스만 조정 가능.

---

## 2.3 스테이지별 “누가 무엇을 완성하고, 어떻게 이어지는지”

| 스테이지 | 게임로직 담당 완성물 | 데이터 담당 완성물 | 연결·동작 보장 |
|----------|----------------------|--------------------|----------------|
| **Phase 0** | 프로젝트·폴더·씬 뼈대 | (없음) | 씬 실행 가능 |
| **Phase 1** | Player 이동·카메라 추적, **PlayerStatsData 참조로 이동 속도 읽기** | `PlayerStatsData` 1개 (moveSpeed 등) | Inspector에서 에셋 연결 → 이동 속도가 데이터 값으로 동작 |
| **Phase 2** | DungeonGenerator, **DungeonSettingsData 참조로 맵 크기·방 개수 읽기** | `DungeonSettingsData` 1개, Wall/Floor 프리팹 | 에셋 연결 시 맵이 설정대로 생성 |
| **Phase 3** | EnemyController, 스폰·전투, **EnemyStatsData 참조** | `EnemyStatsData` N개 (슬라임, 박쇠 등) | 스폰 시 stats 할당 → 체력·데미지가 데이터대로 동작 |
| **Phase 4** | GameManager, UI, 재시작, **플레이어 HP/점수 등 표시** | (필요 시 UI용 데이터) | 게임오버·재시작 시 동작 확인 |
| **Phase 5** | 사운드·이펙트 **호출만** (클립·프리팹은 참조) | BGM·효과음 클립, 이펙트 프리팹, (선택) AttackHitData | 이벤트 시 사운드·이펙트 재생/생성 |

각 스테이지 끝에 **“데이터 에셋을 연결했을 때 기대한 수치/동작이 나오는지”** 한 번씩 확인하면, **이어서 연결 완성**이 보장됩니다.

# Part 3. 스테이지별 동작 보장

## 3.1 “동작 보장”이란

각 개발 스테이지가 끝났을 때,  
- 해당 스테이지 목표가 **완료 조건**을 만족하고,  
- **검증 단계(체크리스트)** 를 통과하면,  
그 스테이지의 동작이 **보장된 것**으로 봅니다.  
이후 스테이지는 “이전 스테이지가 이미 동작한다”는 전제로 이어서 개발합니다.

---

## 3.2 Phase별 완료 조건·검증 (동작 보장 체크리스트)

### Phase 0

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| 프로젝트·폴더·씬 구성 완료 | 프로젝트 열기 → Game 씬 실행 → 빈(또는 최소) 화면 출력 |
| 역할·문서 공유 완료 | 팀원이 본인 직무 문서와 Part 2 인터페이스를 확인했는지 확인 |

**동작 보장**: “플레이 버튼으로 씬이 재생되고, 에러 없음”이면 통과.

---

### Phase 1

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| 플레이어가 입력에 따라 이동한다 | WASD/방향키로 이동, 막힘 없음 |
| 이동 속도가 **데이터(PlayerStatsData)** 에서 나온다 | Inspector에서 moveSpeed 변경 시 빌드에서 속도가 바뀌는지 확인 |
| 카메라가 플레이어를 따른다 | 플레이어 이동 시 화면이 따라오는지 확인 |

**동작 보장**: 데이터 연결 후, “입력 → 이동 → 카메라 추적”이 한 번에 동작하면 통과.

---

### Phase 2

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| 던전이 **데이터(DungeonSettingsData)** 기준으로 생성된다 | roomCountMin/Max 등 변경 시 방 개수·크기가 바뀌는지 확인 |
| 벽/바닥이 배치되고, 플레이어가 벽을 통과하지 않는다 | 맵 안에서 이동, 벽 충돌 확인 |
| 플레이 시마다(또는 설정에 따라) 맵이 다르게 나온다 | 여러 번 플레이 또는 시드 변경으로 구조 변경 확인 |

**동작 보장**: “맵 생성 → 플레이어 스폰 → 이동·충돌 정상”이면 통과.

---

### Phase 3

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| 몬스터가 **EnemyStatsData** 기준으로 스폰·행동·데미지한다 | stats 다른 에셋 할당 시 체력·속도·데미지가 바뀌는지 확인 |
| 플레이어 공격으로 몬스터 사망, 몬스터 접촉 시 플레이어 피격 | 공격 시 적 사망, 적에게 닿으면 HP 감소 확인 |
| (선택) 레벨/경험치가 오른다 | 몬스터 처치 시 EXP 증가·레벨업 확인 |

**동작 보장**: “스폰 → AI → 전투 → 사망/피격”이 데이터대로 동작하면 통과.

---

### Phase 4

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| HP가 UI에 반영된다 | 피격 시 HP 바·숫자 감소 확인 |
| 플레이어 사망 시 게임 오버 패널이 뜬다 | HP 0 → 게임 오버 화면 표시 |
| 재시작 버튼으로 같은 씬이 다시 로드된다 | 재시작 후 맵·플레이어 재생성 확인 |

**동작 보장**: “HP 표시 → 사망 → 게임 오버 → 재시작”이 한 흐름으로 동작하면 통과.

---

### Phase 5

| 완료 조건 | 검증 방법 |
|-----------|-----------|
| 공격/피격/사망 등 이벤트 시 **사운드·이펙트**가 재생/표시된다 | 각 이벤트에서 소리·이펙트 발생 확인 |
| (선택) BGM 재생·게임 오버 시 중지 | BGM 루프, 게임 오버 시 정지 확인 |

**동작 보장**: “이벤트 → 사운드 + 이펙트”가 연결표대로 동작하면 통과.

---

## 3.3 스테이지 진행 순서와 동작 보장 요약

1. **Phase 0** 완료 → 동작 보장 체크 통과 후 **Phase 1** 시작.  
2. **Phase 1** 완료 → 데이터 연결 포함 검증 통과 후 **Phase 2** 시작.  
3. 같은 방식으로 **Phase 2 → 3 → 4 → 5** 순서로 진행.  
4. 각 Phase에서 **게임로직**과 **데이터**는 Part 2의 **인터페이스**에 맞춰 각자 완성한 뒤, **참조만 연결**해 동작을 확인합니다.  
5. **QA**는 위 Phase별 검증을 **체크리스트**로 실행하고, 통과 시 “해당 스테이지 동작 보장 완료”로 기록합니다.

| **게임로직 vs 데이터** | 게임로직은 “동작 코드”, 데이터는 “에셋·수치”; **인터페이스(클래스·필드)** 로만 연결 |
| **연결 완성** | 각 스테이지에서 데이터 에셋을 Inspector에서 연결했을 때, 기대한 수치·동작이 나오면 “연결 완료” |
| **동작 보장** | Phase별 **완료 조건** + **검증 체크리스트**를 통과하면 해당 스테이지 동작 보장, 다음 스테이지 진행 |

이 문서를 기준으로 **직무별 문서·역할을 나누고**, **각 시스템을 단계적으로 완성**하며, **게임로직과 데이터를 분리해 이어서 연결**하고, **스테이지마다 동작이 보장되도록** 진행할 수 있습니다.

**시스템의 점증적 완성**, **스킬·몬스터·아이템을 하나씩 확장하는 내부 구조**는 14번 문서를 참고하세요.

**기획·에셋 제작·코딩**이 **분리된 문서·역할**이면서 **연결되어 확인** 가능하고, **기획 문서 기준**으로 개발·제작이 이루어지며, **기획 보강 시**에도 해당 문서 기준으로 진행할 수 있도록 한 통합 문서는 18번 문서를 참고하세요.
