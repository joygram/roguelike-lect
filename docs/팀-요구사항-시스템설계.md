# 요구사항 분석 및 시스템 설계

기획서(03)와 요구분석(01)에서 정한 내용을 바탕으로 **요구사항 목록**을 정리하고 **시스템 구성**을 설계하는 문서다. 팀이면 기획이 요구사항 표를 채우고, 프로그래밍이 모듈·데이터 흐름을 확인한 뒤 공유하면 된다.

| 폴더·이름 | 이 문서의 모듈(대응) |
|-----------|----------------------|
| **`_Framework/` 기반** | 입력, 플레이어(이동·전투 **규칙 코드**), 전투(판정·데미지 훅), 게임 흐름 |
| **`_Content/` 컨텐츠** | 스테이지·맵(**배치·SO·프리팹**), 적(**스폰·에셋**) |
| **`_UI/`** | HUD·게임오버 — **표시·버튼만**; HP 계산·클리어 판정은 기반 |

**전체 흐름에서의 위치**: 이 문서는 **게임 구조 설계** 단계(요구·모듈·**세 폴더** 배치)의 중심입니다. 코어·컨텐츠·UI **구현** 순서는 루트 README **[네 축 흐름](00-강좌-규약-및-참고.md#dev-flow-axes)** · [07 참고 표](07-참고-정본표-폴더.md).

## 이 단계에서 직접 할 일 (순서대로)

| 순서 | 할 일 | 완료 시 |
|:----:|--------|---------|
| 1 | **1.1 기능 요구사항** 표를 보고, 우리 게임에 **없는 기능**은 비고에 "제외", **추가할 기능**은 새 행으로 넣고 ID(F-15 등)를 부여한다. | 기능 요구사항 확정 |
| 2 | **1.2 비기능 요구사항**에서 우리 팀 목표(성능, 조작, 호환)에 맞게 설명을 수정한다. | 비기능 요구사항 확정 |
| 3 | **1.3 요구사항 추적**: 우리 기획서 항목(예: "2.3 조작")과 요구사항 ID(예: F-01)를 매핑한 표를 한 줄이라도 적어 둔다. | 추적표 1차 작성 |
| 4 | **2.2 모듈 목록**을 읽고, 우리 게임에서 **쓰지 않는 모듈**은 비고에 "미사용", **이름을 바꿀 모듈**은 우리 쪽 이름으로 적어 둔다. | 시스템 구성 확정 |
| 5 | 팀에 요구사항·시스템 설계 문서를 공유한 뒤, 다음 단계(05)로 넘어간다. | 팀 공유 완료 |

## 2. 시스템 설계

### 2.1 시스템 구성도 (개념)

```
[ 입력 ]  →  [ 플레이어 컨트롤러 ]  →  [ 캐릭터 (이동/점프/공격/HP) ]
                  ↑
[ 스테이지·레벨 ] ← [ 맵 데이터 (타일맵·플랫폼) ]
       ↓
[ 적 스폰/관리 ]  →  [ 적 AI / 전투 ]
       ↓
[ 전투 시스템 ]  ←  [ 충돌/데미지 ]
       ↓
[ 게임 매니저 ]  →  [ UI (HP, 점수, 게임오버) ]  →  [ 씬 전환/재시작 ]
```

### 2.2 모듈(시스템) 목록

| 모듈명 | 역할 | 주요 클래스/오브젝트 예시 | 의존성 | **사용 지식·영역** |
|--------|------|---------------------------|--------|--------------------|
| **입력** | 키/버튼 입력 수집 | InputManager, PlayerInput | 없음 | 게임 물리·입력 |
| **플레이어** | 이동, 점프, 공격, HP | PlayerController, PlayerHealth | 입력, 전투 | 게임 물리·입력, 데이터 설계, 소프트웨어 설계 |
| **스테이지·맵** | 스테이지/레벨 로드, 플랫폼 배치 | LevelManager, Tilemap/Platform | 없음 | 데이터 설계, 알고리즘(맵), 소프트웨어 설계 |
| **적** | 스폰, AI, HP, 사망 | EnemySpawner, EnemyAI, EnemyHealth | 스테이지, 전투 | 알고리즘(AI·스폰), 데이터 설계, 소프트웨어 설계 |
| **전투** | 데미지 계산, 충돌 처리 | CombatSystem, DamageHandler | 플레이어, 적 | 알고리즘(판정), 데이터 설계, 게임 물리·입력 |
| **게임 흐름** | 스테이지 시작/종료, 재시작 | GameManager | 플레이어, UI, 씬 | 소프트웨어 설계(상태 관리) |
| **UI** | HP, 점수, 게임오버 화면 | UIManager, HUD, GameOverPanel | 게임 매니저 | UI·사용자 경험, 데이터 설계 |
| **이벤트·트리거** | 글로벌 알림 + 맵/전투 트리거 진입 | `GameEvents`, `MapEventTrigger`, 히트박스 Trigger | 플레이어·맵·전투 | 소프트웨어 설계 — **개발 순서**는 [README #interaction-events](00-강좌-규약-및-참고.md#interaction-events) |

### 2.3 데이터 흐름 (요약)

1. **시작**: GameManager가 스테이지/레벨 로드 → 플랫폼·맵 배치.
2. **스폰**: LevelManager/EnemySpawner가 스테이지 내 위치에 적 배치.
3. **플레이**: Input → PlayerController(이동/점프/공격) → Combat(충돌 시 데미지).
4. **피드백**: PlayerHealth/EnemyHealth 변경 → **글로벌 이벤트(F-16)** 로 UI 갱신(**`_UI/`는 `_Framework/` 이벤트만** 구독). 맵 트리거(F-15)는 **구역 진입 시** 스폰·씬 전환 등.
5. **종료**: PlayerHealth 0 → **GameManager(`_Framework/`)** 가 게임 오버 처리 → **`_UI/`** 표시 → 재시작 시 1번부터.

### 2.4 씬 구성 · 게임 흐름 · 씬 전환

**통합 정본**: 루트 [README `#scene-flow`](00-강좌-규약-및-참고.md#scene-flow) — 씬 다이어그램·Build 순서·`SceneFlow` 예·**씬별 세 폴더 배치**. [32](32-Unity-씬-게임흐름-씬전환.md)는 위 링크 **모음 안내**.

**한 줄 흐름**: 랜딩(선) → 메뉴 → **Game** → (클리어 시 증강·다음 씬) / (사망 시 게임오버 패널 또는 씬) → 재시작·메뉴.

| 씬명(예) | 용도 |
|----------|------|
| **Landing** / Boot | (선택) 앱 시작 직후 로고·자동 다음 씬 — Build **0번** 후보 |
| MainMenu | 타이틀·**시작**·종료 (권장) |
| Game / Stage_* | 메인 플레이 (맵·적·HUD는 보통 동일 씬 또는 스테이지별 씬) |
| AugmentSelect | (선택) 증강 전용 씬 — 없으면 Game 내 UI |
| GameOver | (선택) 전용 씬 — 없으면 Game 씬 내 패널(문서 기본 권장) |

**씬 전환**: **`_Framework/`**(`SceneFlow`·GameManager)에서 `SceneManager.LoadScene` — UI 버튼은 **`_UI/`**에서 기반만 호출. **stageId → 씬 이름**은 **`_Content/` SO**. **Build Settings**에 위 씬들을 **플레이 순서에 맞게** 등록.

### 2.5 프리팹·에셋 의존성

**씬마다 무엇을 둘지**(Landing·MainMenu·Game의 Framework/Content/UI)는 [README `#scene-flow`](00-강좌-규약-및-참고.md#scene-flow) 표와 동일 선상입니다.

- **Player**: Player 프리팹 1종 (스크립트: 이동, 점프, 공격, HP).
- **Enemy**: Enemy 프리팹 1종 이상 (스크립트: AI, HP, 데미지 반응).
- **Map**: 플랫폼·바닥·장애물 프리팹 또는 2D 타일맵 에셋.
- **UI**: 캔버스, HP 바, 게임오버 패널 등.

자세한 에셋 목록은 `05-필요-에셋-목록.md`를 참고하세요.  
**기획 정의 ↔ 클래스·변수명**을 1:1로 맞추고, 기획서를 보고 클래스에 반영하려면 `24-기획-데이터-코드-연결-템플릿.md`를 참고하세요.

## 3. 설계 원칙 (팀 합의 사항)

- **단일 책임**: 한 스크립트는 한 가지 역할(이동, HP, 스폰 등)만 담당.
- **데이터 분리**: 수치(HP, 공격력, 속도)는 Inspector 또는 ScriptableObject로 분리해 기획 변경에 대응.
- **씬 독립성**: 게임 플레이 로직은 가능한 한 특정 씬에 묶지 않고, 씬은 “오브젝트 배치” 용도로 사용.
