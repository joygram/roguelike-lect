# 기획 단계별 작성 — 시스템·컨텐츠·데이터 기획 완성

**기획** 담당이 **단계적으로** 필요한 기획을 모두 완성할 수 있도록,  
**무엇을 어떻게 작성하는지** 순서대로 정리한 문서입니다.  
게임 기획서(GDD), 시스템 기획, 컨텐츠 기획, 데이터 기획을 **순서대로 작성**하면 개발에 바로 넘길 수 있습니다.

---

# Part 0. 기획 개요

## 0.1 기획에서 “완성”이란

- **게임 기획서(GDD)** 가 팀이 합의한 범위를 담고 있고,  
- **시스템 기획**(어떤 모듈이 있고, 어떻게 연결되는지)이 정리되어 있으며,  
- **컨텐츠·데이터 기획**(스킬·적·맵·아이템의 수치·규칙)이 테이블로 나와 있고,  
- **요구사항**이 ID와 함께 정리되어 개발 단계에서 “무엇을 만들지” 추적 가능한 상태를 말합니다.

## 0.2 기획 단계 흐름 (5단계)

| 단계 | 이름 | 산출물 | 완료 시 |
|:----:|------|--------|---------|
| **1** | 게임 기획서(GDD) 작성 | GDD 초안 문서 | 팀이 “이 게임이 뭔지” 공유 완료 |
| **2** | 시스템 기획 | 시스템 목록, 모듈·연결 정의 | “무엇을 만들어야 하는지” 구조 파악 |
| **3** | 데이터·컨텐츠 기획 | 데이터 테이블, 스킬/적/맵 규칙 | 수치·규칙이 개발에 전달 가능 |
| **4** | 요구사항·상세 기획 | 요구사항 목록, 기획서↔ID 매핑 | 개발 Phase별로 “무엇을 구현할지” 명확 |
| **5** | 기획 검토·확정 | 검토 체크리스트, 합의 결과 | 기획 단계 종료, 개발 착수 |

→ 아래 Part 1~5는 **각 단계를 실제로 어떻게 작성하는지** 단계별로 안내합니다.

# Part 1. 1단계 — 게임 기획서(GDD) 작성

## 1.1 목표

- “**우리가 만드는 게임이 무엇인지**”를 한 문서로 정리한다.
- 팀이 같은 목표를 공유하고, 이후 시스템·컨텐츠 기획의 **기준**이 되는 문서를 만든다.
- [03 게임기획서 초안](03-게임기획서-초안.md)을 쓸 때는 상단 **[기획 확인 체크리스트](03-게임기획서-초안.md#gdd-checklist-29)** 를 빠짐없이 채운 뒤 [29](29-게임기획서-확인-체크리스트.md)로 보유/누락을 표시합니다.

## 1.2 산출물

- **게임 기획서(GDD) 초안** 1부 (문서 또는 스프레드시트).

## 1.3 실제 작성 방법

### Step 1-1: 개요 채우기

아래 표를 복사해 **괄호 안을 실제 내용으로** 채운다.

| 항목 | 작성할 내용 | 예시 |
|------|-------------|------|
| 게임명 | 프로젝트에서 부르는 이름 | (예) 플랫폼 점프 대작전 |
| 장르 | 게임 종류 (플랫포머, 액션 등) | 2D 도트 액션 플랫포머 |
| 대상 유저 | 나이·경험 수준 | 기획서에 명시 (티어·과제 범위 등) |
| 플랫폼 | 어떤 기기에서 실행하는지 | PC (Windows/Mac) |
| 엔진 | 사용하는 도구 | Unity 6.3 LTS |
| 개발 기간 | 예상 주·차시 수 | 8~12주 (차시 기준) |
| 참고 레퍼런스 | 비슷한 게임·영상 (선택) | (게임명 또는 URL) |

### Step 1-2: 핵심 플레이 문장으로 정리

다음 네 가지를 **한 문장씩** 쓴다.

1. **플레이어 목표**: “플레이어는 ○○를 해서 △△한다.”  
   예: “플레이어는 스테이지를 진행하며 플랫폼을 점프하고 적을 처치해 목표 지점에 도달한다.”
2. **조작**: “이동은 □□, 점프는 □□, 공격은 □□.”  
   예: “이동은 A/D, 점프는 스페이스, 공격은 Z키.”
3. **승리 조건**: “○○하면 승리.” 또는 “승리 조건 없음, 생존이 목표.”
4. **패배 조건**: “○○하면 게임 오버.”  
   예: “플레이어 HP가 0이 되면 게임 오버 후 재시작.”

### Step 1-3: 스테이지·맵(플랫폼) 규칙 (짧게)

- 스테이지가 **고정 맵인지**, 소수 레벨인지.  
- **플랫폼·바닥·벽** 구성, 점프로 이동 가능한 구역.  
- 예: “고정 스테이지 1~3개, 플랫폼·바닥·벽 타일.”

### Step 1-4: 적 최소 정의

- **적 종류 수**: 최소 1~2종부터.  
- **종류별 한 줄 설명**: 이름, “느리지만 튼튼”, “빠르지만 HP 낮음” 등.  
- 예: “슬라임: 느리게 다가옴, 데미지 낮음. 박쥐: 빠름, HP 낮음.”

### Step 1-5: 플레이어·전투·아이템(선택)

- **플레이어**: HP 있음, 좌우 이동·점프·공격 가능.  
- **전투**: 접촉 또는 공격 시 데미지.  
- **아이템**: 있으면 “어떤 종류(회복, 강화 등)가 있는지” 한 줄. 없으면 “1차에는 없음”으로 명시.

### Step 1-6: 기술·리소스

- **엔진·플랫폼**: 이미 개요에 있으면 생략 가능.  
- **필요한 에셋**: “캐릭터, 맵 타일, UI, 효과음” 정도만 나열.  
- **범위**: “우리 팀 규모로 8주 안에 완성 가능한 범위”인지 한 줄로 확인.

## 1.4 1단계 완료 조건

- [ ] 개요 표가 채워져 있다.  
- [ ] 핵심 플레이(목표·조작·승리·패배)가 문장으로 적혀 있다.  
- [ ] 맵·몬스터·전투·아이템(유무)가 한 줄 이상씩 정리되어 있다.  
- [ ] 팀이 GDD 초안을 한 번 읽고 “이 게임을 만든다”고 합의했다.

# Part 2. 2단계 — 시스템 기획

## 2.1 목표

- 게임이 **어떤 시스템(모듈)** 로 이루어지는지 목록을 만든다.  
- 각 시스템이 **무엇을 담당하고**, **어떤 순서로 연결**되는지 정리해, 개발 시 “무엇을 구현할지” 기준을 만든다.

## 2.2 산출물

- **시스템(모듈) 목록** 1표.  
- **시스템 연결도**(또는 데이터 흐름) 1개 (글로만 써도 됨).

## 2.3 실제 작성 방법

### Step 2-1: “필요한 시스템” 나열

GDD를 보면서 “**이 기능을 담당할 것**”을 하나씩 이름 붙여 나열한다.

| 번호 | 시스템(모듈) 이름 | 하는 일 (한 줄) |
|:----:|-------------------|-----------------|
| 1 | 입력 | 키보드/입력 수집 |
| 2 | 플레이어 | 이동, 공격, HP |
| 3 | 스테이지·맵 | 스테이지(플랫폼·맵) 배치 |
| 4 | 몬스터 | 스폰, AI, HP, 사망 |
| 5 | 전투 | 데미지, 충돌 처리 |
| 6 | 게임 흐름 | 시작·종료·재시작 |
| 7 | UI | HP·점수·게임오버 표시 |

→ 프로젝트에 맞게 **이름을 바꾸거나**, “스킬”, “아이템” 등을 추가해도 된다.

### Step 2-2: “누가 누구를 사용하는지” 연결

- **입력** → 플레이어가 사용.  
- **스테이지·맵** 로드 → 플랫폼·맵 배치 → **적**이 스테이지 내에 스폰.  
- **플레이어·몬스터** → **전투**가 데미지 처리.  
- **플레이어 HP 0** → **게임 흐름**이 게임 오버 처리 → **UI**에 표시.

위와 같이 **화살표**로 “A가 B를 호출한다 / A가 B 데이터를 쓴다”를 5~10줄로 적으면 된다.  
(자세한 그림은 `04-요구사항-및-시스템설계.md` 2.1절 참고.)

### Step 2-3: 시스템별 “주요 산출물” 한 줄

개발 시 “이 시스템이 끝나면 뭐가 나와야 하냐”를 한 줄씩 쓴다.

| 시스템 | 완성 시 나오는 것 (예시) |
|--------|--------------------------|
| 입력 | 키 입력을 플레이어에게 전달 |
| 플레이어 | Player 프리팹, 이동·공격·HP 스크립트 |
| 스테이지·맵 | LevelManager, 타일맵·플랫폼 데이터 |
| 몬스터 | Enemy 프리팹, 스폰·AI·HP 스크립트 |
| 전투 | 데미지 적용 로직, 충돌 처리 |
| 게임 흐름 | GameManager, 씬 전환·재시작 |
| UI | HUD, 게임오버 패널, 재시작 버튼 |

## 2.4 2단계 완료 조건

- [ ] 시스템(모듈) 목록이 1표로 정리되어 있다.  
- [ ] 시스템 간 “누가 누구를 쓰는지” 연결이 글이나 그림으로 있다.  
- [ ] 팀이 “이 시스템들을 개발 단계에서 구현한다”고 확인했다.

# Part 4. 4단계 — 요구사항·상세 기획

## 4.1 목표

- GDD·시스템·컨텐츠를 **요구사항 ID**와 연결해, 개발 Phase별로 “**무엇을 구현할지**”를 추적 가능하게 한다.  
- “기획서 2.3절 = F-04”처럼 **매핑**을 한 표로 남긴다.

## 4.2 산출물

- **요구사항 목록**(ID, 이름, 우선순위, 대응 기획 위치).  
- **기획서 ↔ 요구사항 ID 매핑표** (선택이지만 권장).

| F-04 | 스테이지·플랫폼 맵 | 필수 | GDD 1.3 스테이지·맵, 스테이지 설정 테이블 |

→ “이 요구사항은 **어느 기획 문서·어느 테이블**을 보면 된다”가 보이면 된다.

### Step 4-2: Phase별 “이 Phase에서 만족할 요구사항” 정리

단계별 개발 문서(07)의 Phase와 맞춘다.

| Phase | 만족할 요구사항 ID (예시) |
|-------|---------------------------|
| Phase 1 | F-01 (이동) |
| Phase 2 | F-04 · **03이면 F-12**(다음 스테이지 로드)·**F-15**(장애물·입구 트리거·기믹) |
| Phase 3 | F-02~F-08 · **03이면 F-13**(재화)·**F-16**(저주·클리어·UI 동기화) |
| Phase 4 | F-09, F-10, F-03 · **03이면** 증강 3중1택·클리어 흐름 UI(**F-15**·**F-12** 완성) |
| Phase 5 | **F-14** — BGM+SFX 최소 세트 |

→ “이 단계가 끝나면 **이 ID들이 충족**된다”고 팀이 합의하면 된다.

## 4.4 4단계 완료 조건

- [ ] 요구사항 목록에 ID가 있고, 우선순위·대응 기획이 적혀 있다.  
- [ ] (권장) Phase별로 어떤 요구사항을 만족할지 한 줄씩 정리되어 있다.  
- [ ] 개발 담당이 “Phase 진행 시 이 목록으로 확인하겠다”고 확인했다.

# Part 5. 5단계 — 기획 검토·확정

## 5.1 목표

- 기획 산출물이 **완전한지**, **개발에 넘겨도 되는지** 체크리스트로 점검한다.  
- 팀 합의로 **기획 단계 종료**를 선언하고, 개발 단계로 넘어간다.

## 5.2 산출물

- **기획 검토 체크리스트** 결과 (통과/수정 후 재검토).  
- **기획 확정** 기록 (날짜, “기획 완료, 개발 착수” 합의).

## 5.3 실제 작성 방법

### Step 5-1: 기획 검토 체크리스트 실행

`03-게임기획서-리뷰.md`의 **리뷰 체크리스트**를 그대로 사용한다.  
항목별로 ☐를 채우고, **통과 / 수정 후 재리뷰 / 범위 조정** 중 하나를 선택한다.

- **통과**: 대부분 충족 → 5-2로.  
- **수정 후 재리뷰**: 누락 항목 보완 후 다시 체크.  
- **범위 조정**: 기능·일정을 줄이거나 나누고, 필요 시 1~4단계 일부를 다시 정리.

### Step 5-2: 기획 산출물 목록 최종 확인

아래가 모두 있는지 확인한다.

| 번호 | 산출물 | 확인 |
|:----:|--------|:----:|
| 1 | 게임 기획서(GDD) 초안 | ☐ |
| 2 | 시스템(모듈) 목록·연결 | ☐ |
| 3 | 플레이어·적·스킬·스테이지(·아이템) 테이블 | ☐ |
| 4 | 요구사항 목록(ID·우선순위·대응 기획) | ☐ |
| 5 | (선택) Phase별 요구사항 매핑 | ☐ |

### Step 5-3: 팀 합의 기록

- “기획 단계 완료일: (날짜)”  
- “다음 단계: 개발 Phase 0 착수” (또는 팀이 정한 다음 행동)  
- 위를 짧게 문서나 채팅에 남기면 **기획 확정**으로 본다.

## 5.4 5단계 완료 조건

- [ ] 기획 리뷰 체크리스트가 실행되었고, 결과가 “통과” 또는 “범위 조정 후 통과”이다.  
- [ ] 기획 산출물 5종이 준비되었고, 팀이 확인했다.  
- [ ] “기획 완료, 개발 착수”가 팀 합의로 기록되었다.

# Part 6. 기획 단계 요약·한눈에 보기

| 단계 | 할 일 | 산출물 |
|:----:|-------|--------|
| **1** | GDD 개요·핵심 플레이·맵·몬스터·전투·리소스 채우기 | GDD 초안 |
| **2** | 시스템 목록 작성, “누가 누구를 쓰는지” 연결 | 시스템 목록·연결도 |
| **3** | 플레이어·적·스킬·스테이지(·아이템) 수치·규칙 테이블 작성 | 데이터·컨텐츠 테이블 |
| **4** | 요구사항 ID·우선순위·대응 기획, Phase별 매핑 | 요구사항 목록·매핑 |
| **5** | 리뷰 체크리스트 실행, 산출물 최종 확인, 팀 합의 | 검토 결과·기획 확정 |

이 순서대로 진행하면 **시스템 기획·컨텐츠 기획·데이터 기획**을 포함한 **필요한 모든 기획**을 **단계적으로 완성**할 수 있고, 그 결과를 개발에 바로 넘길 수 있습니다.  
각 단계의 세부 형식은 `03`, `04`, `05`, `08` 번 문서를 참고하면서, 위 **실제 작성 방법**만 따라가면 됩니다.

**기획서를 기준**으로 **보강·개선·추가**할 부분을 내용으로 이해하고 채우려면 21번 문서를 사용하세요.

**기획 문서를 기준**으로 에셋 제작·코딩이 연결되고, **기획 보강** 후에도 해당 문서 기준으로 개발·제작이 가능하도록 한 **역할별 연결·문서 기준 개발**은 18번 문서를 참고하세요.
