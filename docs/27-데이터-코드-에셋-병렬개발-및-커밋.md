# 데이터·코드·에셋 분리 병렬 개발 & 소스 커밋

**목적**: 공속·이속·데미지·이펙트·물리 등 **담당을 나눠도** 같은 규약만 지키면 **나중에 합쳐서 한 게임**으로 동작하게 합니다.  
**담당 지정 시**: 아래 **역할별 가이드**만 보면 됩니다. 합치는 순서·Git 커밋 단계도 함께 정리했습니다.

**연계**: [24-기획-데이터-코드-연결-템플릿.md](24-기획-데이터-코드-연결-템플릿.md)(변수명), [05-필요-에셋-목록.md](05-필요-에셋-목록.md), [06-팀원-역할-분담.md](06-팀원-역할-분담.md), [26-CSharp-코드복사-부록.md](26-CSharp-코드복사-부록.md), [README 세 폴더](../README.md#folder-layers) · [**#folder-depth**](../README.md#folder-depth)(담당별 경로·GameEvents 고정).

**맵·장애물 동시 작업**([README #map-asset-pipeline](../README.md#map-asset-pipeline)): 기획 표 → SO → 아트 → 레벨 → 코드 순서로 담당 분리.

**세 폴더와 같이 쓰기**([README #folder-depth](../README.md#folder-depth)): **데이터**는 **`_Content/`**, **코드**는 **`_Framework/`**, **아트**는 Sprites·프리팹. **UI·Canvas**는 **`_UI/`** 담당이 따로 있으면 그쪽만 수정하게 하면 기반·컨텐츠와 충돌이 적습니다.

# Part 1. 세 갈래 정의

| 갈래 | 담당 예시 | 산출물 위치(팀 규약) | 다른 갈래와 맞출 것 |
|------|-----------|----------------------|---------------------|
| **데이터** | 기획·데이터 담당 | `_Content/Data/*.asset` (또는 `Assets/Data/*.asset`) | **영문 필드명** = 24번 표와 동일 |
| **코드** | 프로그래밍 | `_Framework/**/*.cs` 등 (또는 `Assets/Scripts/**/*.cs`) | 데이터 필드명·에셋 **경로·프리팹 이름**만 참조 |
| **에셋(아트)** | 아트·이펙트 | `Assets/Sprites/`, `_Content/Prefabs/`, `Assets/Audio/` | **합의된 폴더·파일명**에 넣기 |

**원칙**:  
- 데이터는 **숫자·참조(Sprite, AudioClip, GameObject 프리팹)** 만 넣고 로직은 넣지 않음.  
- 코드는 **Inspector/ SO에 드래그한 참조**로 이펙트·사운드를 켜고, 하드코딩 경로 최소화.  
- 에셋은 **이름 충돌 방지**를 위해 `VFX_Hit_Player`, `SFX_Hit_Melee` 같이 접두사 통일.

# Part 2. 기능별 분리 표 (공속·이속·데미지·이펙트·물리)

같은 행의 **데이터 / 코드 / 에셋**을 각 담당이 채우면 합쳐집니다.

## 2.1 이동속도(이속)·점프·중력(물리 일부)

| 구분 | 데이터 담당 | 코드 담당 | 에셋 담당 |
|------|-------------|-----------|-----------|
| **할 일** | `PlayerStatsData`에 `moveSpeed`, `jumpForce` 수치 입력. (선택) `gravityScale` 플레이어 전용 값 | `PlayerController2D`가 SO 또는 Inspector의 `moveSpeed`/`jumpForce` 읽기. Rigidbody2D `gravityScale` 적용 | 플레이어 스프라이트·애니(이속에 맞춘 걸음 속도는 선택) |
| **산출물** | `Data/PlayerStats_Default.asset` | `PlayerController2D.cs` | `Sprites/Player/…` |
| **합치기 조건** | 필드명 반드시 `moveSpeed`, `jumpForce` (24번) | `public PlayerStatsData stats` 또는 SerializeField로 SO 할당 | 없음(숫자만 맞으면 동작) |
| **커밋 단위 예** | `data: player moveSpeed 10, jumpForce 14` | `feat: player read stats from PlayerStatsData` | `art: player placeholder sprite` |

## 2.2 공격속도(공속)·공격력(데미지)

| 구분 | 데이터 담당 | 코드 담당 | 에셋 담당 |
|------|-------------|-----------|-----------|
| **할 일** | `attackCooldown`(초), `attackPower`(정수). 적은 `EnemyStatsData`에 동일 개념 | 공격 입력 시 `Time.time >= _nextAttackTime` 검사 후 히트박스 켜기. `_nextAttackTime = Time.time + stats.attackCooldown` | 공격 모션 스프라이트(길이는 공속과 별개로 먼저 가능) |
| **산출물** | `Data/Enemy_RobotToy.asset` 등 | `PlayerAttackHitbox.cs` 확장 또는 `PlayerCombat.cs` | `Sprites/Player/Attack_*.png` |
| **합치기 조건** | `attackCooldown` = 초 단위(기획서 초당 1번 → 1f) | 데미지는 `attackPower`에서 읽기 | 애니 길이와 쿨이 어색하면 데이터만 조정 |
| **커밋 단위 예** | `data: robot attackCooldown 3s windup 1s note in asset` | `feat: attack uses attackCooldown from SO` | `art: candy sword swing 4 frames` |

## 2.3 데미지 숫자 표현·피격 (로직 vs 이펙트)

| 구분 | 데이터 담당 | 코드 담당 | 에셋 담당 |
|------|-------------|-----------|-----------|
| **할 일** | (선택) `DamageType` enum 값, 치명타 배율 | `TakeDamage(int)` 호출 전후로 **이펙트 스폰** 분기 | **히트 이펙트** 프리팹(파티클/스프라이트 애니) |
| **산출물** | `CombatConstants.asset` (선택) | `DamageFeedback.cs`: `Instantiate(hitVfxPrefab, pos, …)` | `Prefabs/VFX/VFX_Hit_Melee.prefab` |
| **합치기 조건** | — | **프리팹 참조는 SO 또는 Inspector**에 `GameObject hitVfxPrefab` | 프리팹 이름·폴더 팀 표에 등록(05 문서) |
| **커밋 단위 예** | `data: add hitstop duration 0.05` | `feat: spawn VFX on damage event` | `fx: VFX_Hit_Melee prefab` |

## 2.4 이펙트(시각) 일반

| 구분 | 데이터 담당 | 코드 담당 | 에셋 담당 |
|------|-------------|-----------|-----------|
| **할 일** | `EffectEntry`: 키 문자열 + 프리팹 참조를 담는 SO (선택) | 키로 프리팹 찾아 스폰, N초 후 `Destroy` | 각 이펙트 프리팹 + 머티리얼 |
| **산출물** | `Data/EffectLibrary.asset` | `EffectSpawner.cs` | `Prefabs/VFX/*` |
| **합치기 조건** | 키 예: `Hit_Player`, `Death_Enemy`, `Jump_Dust` | `PlayEffect(string key, Vector3 pos)` | 프리팹 루트에 **Particle System** 또는 애니 Sprite |
| **커밋** | `data: EffectLibrary 5 entries` | `feat: EffectSpawner by key` | `fx: death smoke prefab` |

## 2.5 물리(레이어·충돌·Rigidbody)

| 구분 | 데이터 담당 | 코드 담당 | 에셋 담당 |
|------|-------------|-----------|-----------|
| **할 일** | (보통 없음) 레이어 이름만 문서로 고정 | `LayerMask` 필드명 `groundLayer`, `enemyLayer` 등. **Physics2D** 호출 시 마스크 전달 | 타일맵/콜라이더가 올바른 **Layer**에 있는지 씬·프리팹 |
| **산출물** | `Docs/PhysicsLayers.md` 팀 공유(표) | `Physics2D.OverlapBox(..., groundLayer)` | 바닥=Ground, 플레이어=Player, 적=Enemy |
| **합치기 조건** | 레이어 이름 문자열 통일 | 코드의 LayerMask가 Project Settings의 레이어와 일치 | 프리팹 저장 시 Layer 포함됨 |
| **커밋** | `docs: physics layer matrix` | `fix: enemy uses Enemy layer in mask` | `art: platform prefab layer Ground` |

### 물리 레이어·충돌 매트릭스 (팀이 채워 유지)

| Layer | Player | Enemy | Ground | HitboxPlayer | HitboxEnemy |
|-------|--------|-------|--------|--------------|-------------|
| Player | — | (충돌/트리거 팀 결정) | O | — | — |
| Enemy | | — | O | O(데미지) | — |
| Ground | | | — | | |

→ **Edit → Project Settings → Physics 2D** 에서 매트릭스 조정. **데이터 SO가 아니라 프로젝트 설정**이므로, 변경 시 **커밋 메시지**에 `project: physics2d layer matrix` 같이 남기고, 팀에 공지.

# Part 3. 담당자별 “이 문서만” 체크리스트

## 데이터 담당

1. [24번](24-기획-데이터-코드-연결-템플릿.md) 표의 **영문 필드**로만 SO 필드 작성.  
2. 이펙트·사운드를 SO에 넣을 때는 **프리팹/클립 슬롯을 비워 둬도 됨**(코드 담당이 플레이스홀더 연결).  
3. 커밋: `Assets/Data/` 위주, 메시지 접두사 `data:`.

## 코드 담당

1. 수치는 **직접 숫자 박지 말고** `PlayerStatsData`/`EnemyStatsData` 참조 우선.  
2. 이펙트는 `SerializeField GameObject hitVfx` 또는 `EffectLibrary`로만 받기 — **경로 문자열 하드코딩 지양**.  
3. 커밋: `Assets/Scripts/`, 메시지 `feat:` / `fix:`.

## 에셋 담당

1. [05번](05-필요-에셋-목록.md) 표에 **파일명·폴더** 등록 후 그대로 저장.  
2. 프리팹은 **레이어·태그**까지 맞춰 저장.  
3. 커밋: `Assets/Sprites`, `Prefabs/VFX`, `Audio`, 메시지 `art:` / `fx:` / `audio:`.

# Part 4. 결과물 합치는 순서 (머지 순서)

병렬 작업 후 **충돌을 줄이는 머지 순서** 권장:

| 순서 | 브랜치·내용 | 이유 |
|:----:|-------------|------|
| 1 | `main`에 **폴더 구조 + 빈 SO 스키마**(필드만, 값 기본) | 코드·아트가 같은 클래스명·경로를 가정 |
| 2 | **데이터** 브랜치 머지 (`data/*`) | 수치·참조 채움 |
| 3 | **코드** 브랜치 머지 | SO 필드명이 이미 고정됨 |
| 4 | **에셋** 브랜치 머지 | 바이너리 충돌 분리, 프리팹을 SO 슬롯에 연결 |
| 5 | **씬** 마지막 조정 한 커밋 | 한 명이 씬에서 참조 정리 |

**Unity YAML 충돌**: 같은 씬을 두 명이 동시에 수정하면 충돌이 크므로, **씬 담당 1명** 또는 **프리팹 단위로만 배치** 후 씬은 마지막에 합치기.

# Part 5. Git 브랜치·커밋 규칙 (예시)

## 브랜치 이름

| 패턴 | 용도 |
|------|------|
| `data/player-stats` | 플레이어 SO 수치 |
| `code/combat-cooldown` | 공속·히트박스 |
| `art/vfx-hit` | 타격 이펙트 |
| `integrate/phase3-merge` | 위 브랜치들 합치기 전용 |

## 커밋 메시지 접두사

```
data: 플레이어 attackCooldown 1.0
feat: PlayerCombat에서 attackCooldown 적용
fx: VFX_Hit_Melee 프리팹 추가
fix: Ground 레이어 마스크 누락
docs: 27 물리 매트릭스 표 갱신
project: Physics 2D 레이어 충돌 설정
```

## 단계별 워크플로 (팀)

1. `git pull` 후 `git checkout -b data/xxx`  
2. 작업 → `git add` 범위를 **본인 갈래 폴더 위주**로 (Data만, Scripts만, Sprites만).  
3. `git commit -m "data: ..."`  
4. `git push origin data/xxx` → PR → 리뷰 후 `main` 머지.  
5. 다음 담당은 **최신 main**에서 다시 브랜치.

**1인 개발**: 커밋만 위 접두사로 나누어 히스토리를 읽기 쉽게 유지.
