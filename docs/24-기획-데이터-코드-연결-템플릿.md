# 기획·데이터·코드 연결 템플릿

**README 전체 순서**: **5단계** — [29](29-게임기획서-확인-체크리스트.md) **다음**. ([03](03-게임기획서-리뷰.md)보다 **앞**.)

기획서에 **없을 수 있는 시스템 디테일**을 보강하고, **기획 정의 ↔ 변수명·클래스**를 1:1로 맞춰  
**기획서를 보면 클래스에 그대로 반영**할 수 있도록 하는 템플릿입니다.  
아래 표를 채우면 **데이터 준비**가 되고, 코딩 시 **같은 영문 변수명·스트링**을 그대로 사용하면 됩니다.

- **한글**: 기획서·기획 정의용.
- **영문**: 코드 클래스·필드·상수명. 이 이름을 **그대로** 스크립트에 사용합니다.
- **클래스 반영**: 해당 값이 들어갈 ScriptableObject 또는 MonoBehaviour·필드.

**게임기획서초안**([03-게임기획서-초안.md](03-게임기획서-초안.md))에 적힌 수치·항목을 **해석한 예시**를 아래 **「게임기획서초안 → 데이터·코드 반영」** 에 두었습니다. 팀 GDD가 다르면 표의 값만 바꾸면 됩니다. 기획서 확인: [03 초안 기획 확인 체크리스트](03-게임기획서-초안.md#gdd-checklist-29) → [29](29-게임기획서-확인-체크리스트.md) 보유/누락·G5 전 `README` 순서.

**표 → 프로젝트 어디에 두나**([README 세 폴더](../README.md#folder-layers)): 수치·SO → **`_Content/Data`**. 규칙 스크립트 → **`_Framework/`**. UI → **`_UI/`** (기반 이벤트만 구독). 스테이지·`LevelSettingsData`·장애물 플래그는 [README #map-asset-pipeline](../README.md#map-asset-pipeline) · [28](28-기획서-맵구성-에셋-구현-연동.md).

**전체 흐름**: [README 네 축](../README.md#dev-flow-axes)에서 **구조 설계**와 **`_Content/` 데이터** 사이를 잇는 단계입니다.

# Part 4. 코드에서 사용 — 변수명·스트링 그대로 쓰기

## 4.1 ScriptableObject 클래스 정의 (기획 정의와 동일한 필드명)

Part 2 템플릿에 채운 **영문**을 그대로 필드명으로 사용합니다.

```csharp
// PlayerStatsData.cs — Part 2.1 플레이어 데이터와 1:1 대응
using UnityEngine;

[CreateAssetMenu(fileName = "PlayerStats", menuName = "Platformer/Player Stats")]
public class PlayerStatsData : ScriptableObject
{
    [Header("체력")]
    public int maxHp = 100;
    [HideInInspector] public int currentHp;  // 런타임용, 에셋에는 안 넣음

    [Header("이동·점프")]
    public float moveSpeed = 5f;
    public float jumpForce = 12f;
    public float gravityScale = 2f;

    [Header("공격")]
    public int attackPower = 10;
    public float attackCooldown = 0.5f;

    [Header("피격")]
    public float invincibleDuration = 1f;

    [Header("레이어")]
    public string groundLayerName = "Ground";
}
```

## 4.2 스트링 키 상수 (기획 문구와 매칭)

Part 2.5에 정의한 **영문 키**를 그대로 상수명으로 사용합니다.

```csharp
// GameStrings.cs — UI·메시지 스트링 키 (기획서 표시 문구와 매칭)
public static class GameStrings
{
    public const string GameOver_Message = "게임 오버";
    public const string Button_Restart = "재시작";
    public const string UI_HP_Label = "HP";
    public const string StageClear_Message = "클리어!";
}
```

- UI 텍스트에 넣을 때: `textComponent.text = GameStrings.GameOver_Message;`
- 기획서에서 문구만 바꾸면 위 상수 값만 수정하면 됩니다.

## 4.3 MonoBehaviour에서 데이터 참조 (기획 정의 필드 그대로)

```csharp
// PlayerHealth.cs — 기획서 "플레이어 HP" = stats.maxHp / currentHp
public class PlayerHealth : MonoBehaviour
{
    public PlayerStatsData stats;  // Inspector에서 PlayerStats 에셋 할당

    void Start()
    {
        stats.currentHp = stats.maxHp;  // 기획 정의 "최대 체력" → maxHp
    }

    public void TakeDamage(int amount)
    {
        if (stats.currentHp <= 0) return;
        stats.currentHp -= amount;
        // 무적 시간 등은 stats.invincibleDuration 사용
    }
}
```

# Part 5. 요약 — 데이터 준비 흐름

| 순서 | 할 일 | 결과 |
|:----:|------|------|
| 1 | **Part 1**으로 기획서에 없는 시스템 디테일(점프력, 무적시간, 추적거리 등) 확인 | 구현에 필요한 항목 목록 |
| 2 | **Part 2** 템플릿 표를 복사해 프로젝트 문서에 붙이고, 한글·영문·값 채우기 | 기획 정의 ↔ 변수명 1:1 표 |
| 3 | **Part 3**으로 기획서 항목 → 클래스·필드 매핑 정리 | 기획서 보면 어디를 수정할지 파악 |
| 4 | **Part 4**처럼 ScriptableObject·상수 클래스를 **Part 2 영문 필드명 그대로** 정의 | 코딩 시 기획 정의와 동일한 이름 사용 |
| 5 | 기획 수치 변경 시 Part 2 표 수정 → Unity 에셋(Inspector) 또는 상수 값만 수정 | 데이터 준비 유지 |

**한글(기획)** 과 **영문(코드)** 를 한 표에서 함께 관리할 수 있습니다.  
데이터 준비가 된 상태로, 기획서를 보고 클래스에 반영하고 코드에서 그대로 사용하면 됩니다.
