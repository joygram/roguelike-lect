# 코드 문서 작성 형식 (교육용)

이 프로젝트의 **코드(구현) 문서**는 **구현하면서 학습**되도록 쓴다. 구현 개요, 채택 이유, 사용하는 Unity API를 함께 제시하고, 단순 나열·복붙용 설명은 피한다.

---

## 1. 필수 섹션 순서

| 순서 | 섹션 | 목적 |
|:----:|------|------|
| 1 | **왜 이렇게 짜는가 — 핵심 판단 목록** | 구현자가 "왜?"를 직접 말할 수 있어야 할 질문 목록. "이 단계에서 배울 것"이 아닌 **구현 전에 답을 가지고 있어야 할 판단 항목** |
| 2 | **구분하기: 내가 만드는 것 vs Unity(API)가 주는 것** | 표로 정리 — 책임 경계 이해 |
| 3 | **왜 엔진(API)을 이렇게 쓰나** | 2~4문장 — Update, Rigidbody 등 선택 이유 |
| 4 | **구현 개요** | 씬/오브젝트 구조, **스크립트별 역할 요약 표** |
| 5 | **표기 규칙 (코드 주석)** | `[우리 멤버]` `[우리 메소드]` `[Unity가 호출]` `[Unity API]` 표 |
| 6 | **스크립트별 블록** (아래 1-1 참고) | 각 스크립트마다 **개요·채택 이유·SerializeField 이유·API 표·코드** |
| 7 | **단계별 실습** | 오브젝트 준비, 연결, 테스트 바닥·프리팹 등 순서 |
| 8 | **이전·이후 단계와의 연결** | 표로 무엇이 이어지는지 |
| 9 | **동작이 안 될 때** | 번호 목록 체크 |
| 10 | **이번 단계에서 익히는 개념** | 키워드 + 한 줄 설명 |
| 11 | **완료 확인** | 체크리스트 |

### 1-1. 스크립트별 블록 (필수 구성)

**스크립트(또는 주요 컴포넌트) 단위로** 아래를 반복한다. 예: PlayerMove, PlayerJump, CameraFollow.

| 하위 섹션 | 내용 |
|-----------|------|
| **개요** | 역할 한 줄, 부착 위치, 실행 흐름(어디서 무엇을 언제 하는지) |
| **채택한 방식과 이유** | 표로 «선택» «이유» — 대안과 비교(예: GetAxisRaw vs GetAxis, OverlapCircle vs OnCollisionStay) |
| **SerializeField를 쓴 이유** | 필드별로 «기획/밸런스 조절» «씬에서 연결해야 해서» «튜닝용» 등 설명. SerializeField 안 쓴 필드는 “왜 안 썼는지” 한 줄 |
| **사용하는 Unity API** | API명 / 역할 표 |
| **코드** | 파일 경로·부착 위치 + 전체 코드 (주석 표기 적용) |

## 2. 코드 주석 표기 (문서·예제 공통)

독자가 주석만 보고 **우리 코드 vs Unity**를 구분할 수 있게 한다.

| 표기 | 의미 |
|------|------|
| `[우리 멤버]` | 우리가 선언한 필드·프로퍼티 |
| `[우리 메소드]` | 우리가 정의한 함수 |
| `[Unity가 호출]` | `Start`, `Update`, `OnCollisionEnter2D` 등 엔진이 호출 |
| `[Unity API]` | `Physics2D`, `Input`, `Instantiate` 등 엔진 제공 |

---

## 3. 단계별 실습 작성 요령

- **스크립트별**로 먼저 쓴 뒤, **실습 순서**(오브젝트 준비, 컴포넌트 부착, 씬에서 연결)를 번호로 정리한다.
- **삽입 위치**가 중요한 경우: 파일명, 클래스 안/밖, 어떤 `}` 앞/뒤를 표로 제시.
- **왜 이 순서인지** 한 줄 (예: `Awake`에서 `_rb`를 먼저 받아야 `Update`에서 씀).
- 긴 개념은 **별도 소제목**으로 풀기 (예: `GetAxisRaw` vs `GetAxis`, `linearVelocity` vs `velocity`).

## 4. 다른 코드 문서에 적용할 때

- `04-전투-코드.md`, `05-UI-코드.md` 등 **코드.md** 파일은 이 형식을 점진적으로 맞춘다.
- **기획.md / 에셋.md**는 이 문서 전체를 복사할 필요는 없으나, **「이 단계의 목표」「왜 이렇게 설계했는지」**는 동일하게 풍부하게 쓴다.
