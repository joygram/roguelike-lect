### 1.3 설치·프로젝트 생성 절차 (직접 따라 할 수 있게)

**Unity 설치 (처음 한 번만)**

1. 브라우저에서 [unity.com/download](https://unity.com/download) 접속 후 **Unity Hub** 다운로드.
2. Unity Hub 설치 후 실행 → **설치** 탭 → **에디터 추가** → **Unity 6** → **6.3 LTS** 선택 후 설치.
3. (선택) 같은 화면에서 **Windows Build Support** 또는 **Mac Build Support** 모듈 추가.
4. **스크립트 에디터**: Unity 설치 시 **Visual Studio** 또는 **VS Code** 함께 설치 옵션이 있으면 선택.
   - 이미 있으면 Unity 실행 후 **Edit → Preferences → External Tools**에서 에디터 지정.

**프로젝트 생성 (매 프로젝트)**

1. Unity Hub → **새로 만들기** → **2D** 템플릿 선택 (플랫포머 권장).
2. **프로젝트 이름**: 영문으로 입력 (예: MyPlatformer).
3. **위치**: 저장할 폴더 선택 후 **만들기**.
4. 에디터가 열리면 **File → Save As**로 첫 씬을 `Assets/Scenes/Game.unity` 등으로 저장.

### 1.4 Unity 버전 통일 (협업 시 — 동시에 받지 않아도 되는 방법)

팀원이 **각자 다른 시점에** 참여해도, **한 사람이 정한 버전**에 나머지가 맞추면 됩니다. 버전이 다르면 프로젝트가 안 열리거나 씬·프리팹이 깨질 수 있으므로, 아래 **3단계**만 지키면 됩니다.

#### 버전 통일 3단계

| 순서 | 누가 | 할 일 |
|:----:|------|--------|
| **1** | **한 명(먼저 만드는 사람)** | Unity 6.3 LTS로 프로젝트 생성 → `ProjectSettings/ProjectVersion.txt` **첫 줄**을 연다. `m_EditorVersion: 6000.0.30f1` 같은 **숫자**를 복사해 **팀 채팅·문서에 공유**한다. |
| **2** | **나머지 팀원** | Hub → **설치** 탭 → **에디터 추가** → **Unity 6** → **6.3 LTS**에서 **공유받은 번호와 같은 패치**를 설치한다. (없으면 [Unity 릴리스](https://unity.com/releases/release-overview) 참고) |
| **3** | **모두** | Hub **프로젝트** 탭에서 이 프로젝트를 **그 버전**으로 연결한 뒤 연다. (프로젝트 추가 후 연필 아이콘에서 버전 선택) |

**한 줄 요약**: 프로젝트 만든 사람이 `ProjectVersion.txt` 첫 줄 번호 공유 → 나머지는 Hub에서 그 번호와 같은 에디터 설치 → 프로젝트 열 때 그 버전 선택.

**직접 채우기 — 우리 팀 Unity 버전 (한 번 정한 뒤 팀에 공유)**

| 항목 | 우리 팀에서 쓸 값 |
|------|-------------------|
| **에디터 버전 (정확한 번호)** | (예: 6000.0.30f1 — `ProjectSettings/ProjectVersion.txt` 첫 줄) |

### 1.5 Git·GitHub 협업 (포크·담당별 작업으로 충돌 줄이기)

팀이 **한 저장소(레포)**를 기준으로 작업할 때, **같은 파일을 동시에 수정**하면 push/merge 시 **충돌**이 난다.
**담당 영역을 나누고**, 가능하면 **담당자끼리 같은 파일을 덜 건드리게** 하면 충돌을 크게 줄일 수 있다.
아래는 **GitHub 기준**으로, 포크 또는 브랜치를 쓸 때 따라 할 수 있는 절차입니다.

#### 1.5.1 팀 저장소 하나로 통일 (포크 없이 브랜치만 쓸 때)

| 순서 | 할 일                                         | 설명                                                                                                                                                              |
| :--: | --------------------------------------------- | ----------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|  1  | **팀 공용 저장소 1개 정하기**           | 팀장 또는 한 명이 GitHub에 레포를 만들고, 나머지는 그 레포를**clone**해서 사용. (포크는 "내 계정에 복사본"이므로, 팀이 하나의 레포만 쓸 때는 clone이면 됨.) |
|  2  | **main(또는 master) 보호**              | 팀 합의:`main`에는 직접 push하지 않고, **브랜치**를 만들어 작업한 뒤 **Pull Request(PR)**로 합친다.                                                       |
|  3  | **담당별 브랜치 또는 담당별 폴더 규칙** | 아래 "담당별로 건드리는 파일 나누기" 참고.                                                                                                                        |

#### 1.5.2 포크로 작업할 때 (교사·팀장 레포 → 각자 fork)

| 순서 | 할 일                                  | 설명                                                                                                                                                                  |
| :--: | -------------------------------------- | --------------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|  1  | **원본(upstream) 저장소 정하기** | 교사 또는 팀장이 만든 레포를 "원본"으로 한다. 주소 예:`https://github.com/teacher/roguelike-project`                                                                |
|  2  | **각자 포크(Fork)**              | GitHub에서 원본 레포 들어간 뒤**Fork** 버튼 → 내 계정에 복사본이 생긴다.                                                                                       |
|  3  | **내 포크를 내 PC에 clone**      | `git clone https://github.com/내계정/roguelike-project`                                                                                                             |
|  4  | **원본을 remote로 추가 (선택)**  | `git remote add upstream https://github.com/teacher/roguelike-project` → 나중에 원본 최신 내용을 가져올 때 `git fetch upstream` 사용.                            |
|  5  | **작업은 내 포크의 브랜치에서**  | `git checkout -b feature/player-movement` 처럼 브랜치 만들어 작업 후, **내 포크**에 push. 그다음 **원본 레포로 Pull Request**를 보내 팀장/교사가 merge. |

#### 1.5.3 담당자끼리 충돌이 안 나게 — 담당별로 건드리는 파일 나누기

**같은 파일을 두 명이 동시에 수정하면** Git이 "충돌"로 표시한다. **가능한 한 담당 영역을 나누어** 서로 다른 파일/폴더를 수정하게 하면 충돌이 줄어든다.

| 담당 역할 예시            | 주로 수정하는 폴더/파일                           | 다른 담당과 겹치지 않도록                                        |
| ------------------------- | ------------------------------------------------- | ---------------------------------------------------------------- |
| **플레이어 담당**   | `Scripts/Player/`, `Prefabs/Player.prefab`    | 플레이어 씬 배치만 하고, Enemy·Dungeon 스크립트는 건드리지 않기 |
| **적·AI 담당** | `Scripts/Enemy/`, `Prefabs/` 내 적 프리팹 | Enemy 폴더·적 프리팹만 수정                                 |
| **스테이지·맵 담당** | `Scripts/Level/` 또는 `Scripts/Stage/`, 맵 관련 씬·타일맵 | 스테이지·플랫폼 맵 오브젝트만 수정                                 |
| **UI 담당**         | `Scripts/UI/`, UI 캔버스·씬 내 UI              | UI 스크립트·UI 오브젝트만 수정                                  |
| **기획·데이터**    | `Data/` (ScriptableObject 등), 문서(`docs/`)  | 데이터·설정 파일만 수정                                         |

**공통으로 건드릴 수 있는 파일** (충돌 가능성 있음 → 조율 필요):

- `Scenes/Game.unity` (한 명이 씬 레이아웃 담당하고, 나머지는 자기 프리팹만 올리기),
- `ProjectSettings/` (보통 팀장만 변경),
- 루트의 `README.md` (변경 전 팀에 말하기).

**직접 채우기 — 우리 팀 저장소·규칙**

| 항목                       | 우리 팀에서 쓸 값                                      |
| -------------------------- | ------------------------------------------------------ |
| **공용 저장소 주소** | (예: https://github.com/팀명/프로젝트명)               |
| **방식**             | ☐ 팀 레포 1개 + 브랜치 / ☐ 포크 후 PR                |
| **Unity 버전**       | (1.4에서 정한 버전과 동일하게)                         |
| **담당별 폴더**      | 플레이어: Scripts/Player 등 (06 역할 분담과 맞춰 적기) |

😄 **한줄 조크**: 체크리스트가 길어 보여도, **하나씩 체크해 나가면** 어느새 "준비 완료"가 됩니다.

## 3. 에셋(그래픽·소리) 준비

에셋은 **2D 스프라이트 제작·사용**을 기준으로 합니다. **그림을 직접 그리고 Unity에 넣어 보는 과정**을 체험할 수 있고, 픽셀 단위로 캐릭터·맵을 구분하기 쉬우며, 2D 플랫포머에 맞는 작업량으로 맞추기 위해 **스프라이트**를 기준으로 했습니다.  
리소스가 아직 없을 때는 프로그래머가 **플레이스홀더로 먼저 작업한 뒤 나중에 교체**하는 방식으로 진행합니다. → [06 문서 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체)

### 3.1 스프라이트 제작·사용 방식 (기준)

#### 제작 → 저장

| 단계 | 할 일                  | 설명                                                                                                                                                          |
| :--: | ---------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------- |
|  1  | **그리기**       | 그림 프로그램(그림판, Aseprite, Piskel, Photopea 등)에서**투명 배경**으로 캐릭터·적·플랫폼·바닥·아이템을 그린다. 한 캐릭터당 1장 또는 방향별 2~4장. |
|  2  | **저장**         | **PNG**로 저장 (투명 영역 유지). 파일명은 영문 권장 (예: `player_idle.png`, `slime.png`).                                                           |
|  3  | **Unity에 넣기** | 프로젝트의 `Assets/Sprites/` 폴더에 PNG 파일을 **복사**하거나 Unity 에디터에서 해당 폴더로 **드래그 앤 드롭**.                                  |

#### Unity에서 스프라이트로 쓰기

| 단계 | 할 일                             | 설명                                                                                                                                                                                                         |
| :--: | --------------------------------- | ------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------------ |
|  4  | **Texture Type을 Sprite로** | Project 창에서 넣은 PNG를 클릭 → Inspector에서**Texture Type**을 **Sprite (2D and UI)**로 바꾼다. 여러 칸이 한 이미지에 있으면 **Sprite Mode**를 **Multiple**로 두고 **Apply**.     |
|  5  | **Pixels Per Unit (PPU)**   | 같은 Inspector에서**Pixels Per Unit**을 팀에서 정한 값으로 맞춘다. 예: **32** 또는 **64** (타일 1칸 = 1유닛으로 맞출 때). 전체 프로젝트에서 **한 값으로 통일**하면 크기 비율이 맞다. |
|  6  | **씬에서 사용**             | 스프라이트를 Hierarchy 또는 씬 뷰로 드래그하면 Sprite Renderer가 붙은 오브젝트가 생긴다. 또는 빈 오브젝트에**Sprite Renderer**를 추가하고 **Sprite** 칸에 이미지를 넣는다.                       |

**직접 채우기 — 우리 팀 스프라이트 규칙**

| 항목                                 | 우리 팀에서 쓸 값                     |
| ------------------------------------ | ------------------------------------- |
| **Pixels Per Unit (PPU)**      | (예: 32 또는 64 — 한 번 정하면 통일) |
| **캐릭터 기본 크기 (픽셀)**    | (예: 32×32)                          |
| **타일(벽·바닥) 크기 (픽셀)** | (예: 16×16 또는 32×32)              |

### 3.2 필요한 스프라이트 종류·크기 (우리 게임용)

| 용도          | 준비할 것                    | 권장 크기(픽셀)    | 비고                                    |
| ------------- | ---------------------------- | ------------------ | --------------------------------------- |
| 플레이어      | PNG 1장 또는 방향별 2~4장    | 32×32 ~ 64×64    | Sprite Mode: Single 또는 Multiple(시트) |
| 적            | 종류별 1장 이상              | 32×32             | 02 직접 채우기 표의 적 수만큼           |
| 벽·바닥      | 타일 1장 또는 타일셋 여러 장 | 16×16 또는 32×32 | Tilemap 사용 시 타일 크기와 맞추기      |
| 아이템        | 아이콘 1장씩                 | 16×16 ~ 32×32    | 포션, 코인 등 종류별                    |
| UI (HP 바 등) | 배경·충전용 이미지          | 자유               | UI Image에서 Sprite로 지정              |

**저작권**: 교육·팀 프로젝트용은 **직접 제작** 권장. 외부 사용 시 Unity Asset Store 무료, itch.io 교육용 명시 자료만 사용.

---

**스프라이트를 기준으로 한 이유**: **직접 그려 보고 임포트해 보는 과정**을 통해 제작 흐름을 익히고, 2D 로그라이크에 맞는 비주얼을 단계적으로 넣기 위함입니다. 리소스 없을 때 플레이스홀더로 작업 후 교체하는 방법은 [06 문서 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체) 참고.

## 4. 폴더 구조 (프로젝트 내)

미리 만들어 두면 개발 중 “어디에 저장하나요?” 질문을 줄일 수 있습니다. **팀 규약**은 루트 [README 세 폴더 표](00-강좌-규약-및-참고.md#folder-layers)와 **[#folder-depth](00-강좌-규약-및-참고.md#folder-depth)**(체크리스트·GameEvents·경계)·[04](04-요구사항-및-시스템설계.md)·[06](06-팀원-역할-분담.md)·[07 참고 표](07-참고-정본표-폴더.md)와 **같은 규약**입니다. 인쇄용 한 파일: [31](31-프레임워크-컨텐츠-UI-구분-및-구현.md).

### 4.0 권장 — 기반(`_Framework/`) · 컨텐츠(`_Content/`) · UI(`_UI/`)

```
Assets/
├── _Framework/          ← 기반: 씬 전환·런 상태·전투 훅 등 (Runtime/ 등)
├── _Content/            ← 컨텐츠: ScriptableObject, 스테이지 씬, 맵·적·장애물 프리팹
├── _UI/                 ← UI: Canvas, HUD, 메뉴 프리팹
├── Sprites/             ← 아트 공통 (컨텐츠 프리팹·UI Image가 참조)
├── Audio/               ← BGM·SFX
└── (선택) Materials/
```

스테이지 씬은 `_Content/Scenes/` 등 팀이 정한 위치에 두면 됩니다.

### 4.1 단순 구조 (레거시 — 가능하면 위와 병행해 점진 이전)

```
Assets/
├── Scenes/
├── Scripts/         ← Player, Enemy, Level, UI 하위 폴더
├── Prefabs/
├── Sprites/
├── Materials/       (선택)
└── Audio/
```

**폴더 만드는 절차 (직접 따라 할 수 있게)**

1. Unity 에디터에서 **Project** 창을 연다.
2. **Assets** 우클릭 → **Folder**로 **`_Framework`**, **`_Content`**, **`_UI`** 를 만든다. (`_Content` 안에 `Data`, `Prefabs`, `Scenes` 등 하위 폴더는 07 Phase 0·[README #folder-depth](00-강좌-규약-및-참고.md#folder-depth)에 맞춤.)
3. 같은 방식으로 `Sprites`, `Audio` 폴더를 만든다.
4. (선택) 위 **4.1**처럼 `Scripts`·`Scenes`·`Prefabs`만 쓰는 팀은, 담당을 **코어 스크립트 / 프리팹·Data·스테이지 씬 / UI 프리팹**으로만 나눠도 되나, **신규는 _Framework·_Content·_UI 생성을 권장**합니다.
5. (선택) 2D만 쓸 경우 `Materials`는 나중에 만들어도 된다.
6. **File → Save Project**로 저장한다.

**G10 직후**: [README #folder-depth](00-강좌-규약-및-참고.md#folder-depth)에서 기반·컨텐츠·UI·`GameEvents`·I1~I5 합의. **씬 이름·Build 순서·메뉴→Game 전환**은 [README #scene-flow](00-강좌-규약-및-참고.md#scene-flow) · 아래 §4.2와 동일.

### 4.2 씬 이름·에셋 이름 제안 (그대로 써도 됨)

팀에서 이름을 따로 정하지 않았다면 아래 **정확한 이름**을 그대로 사용하면 됩니다. 코드·문서 예시와 일치해서 따라 하기 쉽습니다.

#### 씬 (Scenes 폴더)

**전체 흐름·전환·Build·씬별 프리팹**: [README #scene-flow](00-강좌-규약-및-참고.md#scene-flow) · [32](32-Unity-씬-게임흐름-씬전환.md)(위치 안내).

| 용도 | 정확한 씬 이름 (파일명) | 비고 |
|------|------------------------|------|
| 랜딩·부트 | `Landing.unity` | 선택. Build 0번 후보 → 자동으로 MainMenu 등으로 전환 |
| 타이틀·메뉴 | `MainMenu.unity` | 권장. 시작 버튼으로 Game 로드 |
| 메인 플레이 | `Game.unity` | 필수. File → Save As로 저장 시 이 이름 권장 |
| 스테이지 분리 시 | `Stage_1_1.unity` 등 | [README #map-asset-pipeline](00-강좌-규약-및-참고.md#map-asset-pipeline) · [28](28-기획서-맵구성-에셋-구현-연동.md) |
| 게임 오버 | `GameOver.unity` | 선택. 없으면 Game 씬 안에서 UI로 처리 |
| 증강 전용 | `AugmentSelect.unity` | 선택. 없으면 Game 씬 내 패널 |

#### 프리팹 (Prefabs 폴더)

| 용도 | 정확한 프리팹 이름 | 비고 |
|------|-------------------|------|
| 플레이어 | `Player.prefab` | 필수 |
| 적 (종류별) | `Enemy_Slime.prefab`, `Enemy_Bat.prefab` … | Enemy 폴더에 넣을 때: `Prefabs/Enemies/Enemy_Slime.prefab` |
| 플랫폼·벽 | `Platform.prefab`, `Wall.prefab` | 타일맵 또는 프리팹 배치 |
| 바닥 | `Floor.prefab` | 타일맵 또는 프리팹 배치 |
| (선택) 투사체 | `Projectile.prefab` | 원거리 공격 시 |

#### 데이터 에셋 (Data 폴더, ScriptableObject)

| 용도 | 정확한 에셋 이름 | 비고 |
|------|-----------------|------|
| 플레이어 수치 | `PlayerStats.asset` | Create → Roguelike → Player Stats로 생성 |
| 적 수치 (종류별) | `SlimeStats.asset`, `BatStats.asset` … | `Data/EnemyStats/` 폴더에 두면 정리하기 좋음 |
| 스테이지/레벨 설정 | `LevelSettings.asset` | 맵 크기·스테이지 수 등 |
| (선택) 아이템 | `RedPotion.asset` … | ItemData 형식일 때 |

#### 스프라이트 (Sprites 폴더, PNG 파일명)

| 용도 | 정확한 파일명 예시 | 비고 |
|------|-------------------|------|
| 플레이어 | `player_idle.png` 또는 `player.png` | 영문 소문자·밑줄 권장 |
| 적 | `slime.png`, `bat.png` … | 종류별 1장 |
| 벽·바닥 타일 | `tile_wall.png`, `tile_floor.png` | 타일맵 또는 프리팹 Sprite로 사용 |
| 아이템 아이콘 | `item_red_potion.png` … | |
| UI | `ui_hp_bar.png`, `ui_hp_fill.png` | HP 바 배경·충전용 |

**이름 규칙 요약**: 영문 사용, 공백 대신 밑줄(`_`), 소문자 권장. 씬·프리팹·에셋은 **팀에서 한 번 정한 뒤 문서에 적어 두면** 코드에서 `LoadScene("Game")`, `Instantiate(floorPrefab)` 등으로 일관되게 쓸 수 있습니다.
