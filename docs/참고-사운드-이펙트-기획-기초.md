# 사운드·이펙트 기획 및 자료구조 기초

**사운드**와 **이펙트**의 **기획**이란 무엇인지, **자료구조**로 어떻게 표현하는지에 대한 **상세한 기초 설명**을 담은 문서입니다.  
용어와 구조를 단계별로 정리했습니다.

---

# Part 1. 사운드 기초

## 1.1 사운드가 뭔가요?

게임에서 **들리는 소리**를 통틀어 사운드라고 합니다.  
- **BGM(배경음악)**: 맵이나 메뉴에서 계속 흐르는 음악.  
- **효과음(SFX)**: 공격, 피격, 아이템 획득처럼 **특정 순간**에 한 번 재생되는 짧은 소리.

## 1.2 Unity에서 사운드를 쓰는 기본 단위

| 용어 | 설명 |
|------|-----------|
| **AudioClip** | **소리 파일 하나**를 Unity가 읽은 것. WAV, MP3, OGG 파일을 프로젝트에 넣으면 AudioClip이 됩니다. “재생할 소리의 내용”을 담는 **데이터**입니다. |
| **AudioSource** | **그 소리를 실제로 재생하는 장치**입니다. 스피커처럼 “여기서 소리가 나온다”라고 생각하면 됩니다. 오브젝트에 AudioSource를 붙이면, 그 오브젝트 위치에서 소리가 납니다(공간 음향 사용 시). 2D 게임에서는 보통 2D 설정을 씁니다. |
| **PlayOneShot** | AudioSource에서 “이 클립을 **한 번** 재생해라”라고 할 때 쓰는 방법입니다. 짧은 효과음을 여러 번 겹쳐 재생할 때 유리합니다. |

→ **기획·자료구조**에서 다루는 것은 대부분 “**어떤 순간에 어떤 AudioClip을 재생할지**”입니다.

## 1.3 BGM과 효과음의 차이 (기획 시 구분)

| 구분 | BGM | 효과음(SFX) |
|------|-----|-------------|
| **길이** | 보통 30초~몇 분, **반복(루프)** 가능 | 보통 0.1초~2초, 한 번만 재생 |
| **재생 시점** | 씬 시작, 스테이지 진입 등 **상황 전환** 시 | 공격, 피격, 획득 등 **이벤트** 발생 시 |
| **동시 재생** | 보통 1개만 (BGM 채널 1개) | 여러 개 동시 가능 (공격음+피격음 등) |
| **자료로 쓸 때** | “이 씬/상황의 BGM 클립 1개” | “이벤트별로 재생할 클립 1개씩” |

# Part 2. 사운드 기획 — 무엇을 정하는지

## 2.1 사운드 기획이란

**어떤 상황에서 어떤 소리를 낼지**를 미리 표로 정리하는 것입니다.  
기획 단계에서 정해 두면, 에셋 제작(어떤 파일을 준비할지), 코딩(어디서 재생할지)이 쉬워집니다.

## 2.2 기획할 항목 (사운드)

| 항목 | 설명 | 예시 |
|------|------|------|
| **재생 시점(이벤트)** | 소리를 내는 **게임 상황** | “플레이어 공격이 적에게 닿을 때” |
| **종류** | BGM인지, 효과음인지 | BGM / SFX |
| **클립 식별** | 나중에 쓸 **이름 또는 ID** | attack_hit, hurt, bgm_dungeon |
| **특성** | 루프 여부, 대략 길이 | BGM: 루프 / 효과음: 0.5초 내외 |
| **볼륨(선택)** | 다른 소리 대비 크기 | 보통 1.0, 필요 시 0.7 등 |

## 2.3 사운드 기획표 (예시)

| 이벤트 | 종류 | 클립 식별(이름) | 비고 |
|--------|------|-----------------|------|
| 게임 시작 / 던전 진입 | BGM | bgm_dungeon | 루프 |
| 플레이어 공격 타격 | SFX | sfx_attack_hit | 짧게 |
| 플레이어 피격 | SFX | sfx_hurt | 짧게 |
| 몬스터 사망 | SFX | sfx_enemy_death | 짧게 |
| 아이템 획득 | SFX | sfx_get_item | 짧게 |
| 게임 오버 | SFX | sfx_game_over | 1회, BGM 정지 후 |

→ 이 표가 **사운드 기획** 결과물입니다. 이걸 기준으로 **오디오 파일**을 준비하고, **코드에서 재생 시점**을 연결합니다.

# Part 3. 사운드 자료구조 — 데이터로 어떻게 표현하는지

## 3.1 “자료구조”가 뭔가요?

**자료구조**는 “정보를 어떤 형태로 묶어서 저장하고 찾을지”를 정한 **구조**입니다.  
사운드에서는 “**이 이벤트일 때 이 클립**”을 프로그램이 쓸 수 있게 **변수·테이블·클래스**로 표현하는 것이 자료구조입니다.

## 3.2 가장 단순한 구조: 이벤트별 변수

**한 이벤트당 클립 하나**를 변수로 두는 방식입니다.

```
[구조]
- attackHitClip  (AudioClip)  → 공격 타격 시 재생
- hurtClip      (AudioClip)  → 피격 시 재생
- enemyDeathClip (AudioClip)  → 몬스터 사망 시 재생
```

- **장점**: 이해하기 쉽고, 코드에서 `Play(attackHitClip)` 처럼 바로 씁니다.  
- **단점**: 이벤트가 늘어날 때마다 변수가 늘어납니다.

## 3.3 조금 더 정리한 구조: 리스트(배열) + 이름 매칭

**효과음 목록**을 배열(또는 리스트)로 두고, “이름(문자열)” 또는 “번호”로 찾아 재생하는 방식입니다.

```
[구조]
- sfxClips (리스트 또는 배열)
  [0] → 공격 타격용 클립
  [1] → 피격용 클립
  [2] → 몬스터 사망용 클립
  ...

재생 시: "attack_hit"이면 0번, "hurt"이면 1번 클립 재생
```

- **자료구조**: `List<AudioClip>` 또는 `AudioClip[]`, 그리고 “이벤트 이름 → 인덱스” 매핑(표 또는 switch문).  
- **장점**: 클립만 추가하면 되고, 코드는 “이벤트 이름 → 클립”만 바꾸면 됩니다.

## 3.4 데이터 에셋으로 쓰는 구조: ScriptableObject

Unity에서는 **ScriptableObject**로 “이벤트별로 재생할 클립”을 **에셋**으로 만들 수 있습니다.  
기획표의 **한 행**을 **에셋 하나** 또는 **클래스 필드**로 두는 방식입니다.

```
[구조 예시: SoundEventData (ScriptableObject)]
- eventId (문자열)  → "attack_hit", "hurt" 등
- clip (AudioClip)  → 재생할 소리
- volume (숫자)    → 볼륨 0~1 (선택)
```

- **자료구조**: “이벤트 ID + 클립 + (선택)볼륨”을 묶은 **데이터 타입** 하나.  
- **장점**: 기획에서 “attack_hit용 클립만 바꿔 끼우기”가 쉽고, 에셋으로 관리할 수 있습니다.

## 3.5 BGM용 자료구조

BGM은 **씬·상황당 하나**만 두는 경우가 많습니다.

- **가장 단순**: 변수 하나 `bgmClip (AudioClip)`, 씬 시작 시 재생.  
- **씬별로 다르게**: “씬 이름 → BGM 클립” 매핑 테이블(딕셔너리 또는 ScriptableObject 리스트).

# Part 4. 이펙트 기초

## 4.1 이펙트가 뭔가요?

게임에서 **눈에 보이는 연출** 중, “잠깐 나타났다가 사라지는 것”을 보통 이펙트라고 부릅니다.  
예: 공격이 닿을 때 번쩍임, 피격 시 빨간 깜빡임, 몬스터 사망 시 연기·파티클.

## 4.2 Unity에서 이펙트를 쓰는 기본 단위

| 용어 | 설명 |
|------|-----------|
| **GameObject** | 씬에 있는 **오브젝트 하나**. 빈 오브젝트이거나, 큐브·스프라이트·파티클을 자식으로 가질 수 있습니다. |
| **Prefab(프리팹)** | **오브젝트 설계도를 저장해 둔 것**. 이펙트는 “이펙트용 프리팹”을 만들어 두고, 재생할 때 그 프리팹을 **복사해서 씬에 생성(Instantiate)** 합니다. |
| **Instantiate** | 프리팹을 **복사해서 씬에 하나 만드는** Unity API. “이 위치에 이펙트 프리팹 하나 생성”할 때 씁니다. |
| **Particle System** | 연기·불꽃·반짝임처럼 **작은 입자들이 움직이는** 연출. Unity 기본 컴포넌트입니다. |
| **Sprite** | 2D 이미지 한 장. 이펙트를 “이미지 한 장을 잠깐 보였다가 없애기”로 만들 수도 있습니다. |

→ **기획·자료구조**에서 다루는 것은 “**어떤 순간에 어떤 이펙트 프리팹을, 어디 위치에 생성할지**”입니다.

## 4.3 이펙트의 “생명”

이펙트는 **한 번 생성되면**, 보통 **몇 초 후에 자동으로 사라지게** 합니다.  
- **지속 시간(lifeTime)**: 0.3초, 0.5초 등.  
- **구현**: 프리팹에 “lifeTime 초 후에 자기 자신을 Destroy”하는 스크립트를 붙입니다.  
→ 그래서 **자료**에는 “**어떤 프리팹**을 쓸지”와 (선택) “**지속 시간**”이 들어갑니다.

# Part 5. 이펙트 기획 — 무엇을 정하는지

## 5.1 이펙트 기획이란

**어떤 상황에서 어떤 보이는 연출을, 어디에 보여 줄지**를 표로 정리하는 것입니다.

## 5.2 기획할 항목 (이펙트)

| 항목 | 설명 | 예시 |
|------|------|------|
| **발생 시점(이벤트)** | 이펙트를 띄우는 **게임 상황** | “플레이어 공격이 적에게 닿을 때” |
| **이펙트 종류(이름)** | 기획·에셋에서 부르는 이름 | Effect_Hit_Normal, Effect_Hurt |
| **생성 위치** | 월드 좌표 기준 **어디에** 만들지 | 적 위치, 플레이어 위치, 공격 끝점 |
| **지속 시간** | 몇 초 후 제거할지 | 0.3초, 0.5초 |
| **연결 대상** | 어떤 스킬/시스템에 쓰는지 | 기본 공격 SkillData, PlayerHealth 피격 |

## 5.3 이펙트 기획표 (예시)

| 이벤트 | 이펙트 이름 | 생성 위치 | 지속 시간(초) | 비고 |
|--------|--------------|------------|---------------|------|
| 플레이어 공격 타격 | Effect_Hit_Normal | 적이 맞은 위치 | 0.3 | 기본 공격 |
| 플레이어 공격 타격(강타) | Effect_Hit_Heavy | 적이 맞은 위치 | 0.5 | 스킬: 강타 |
| 범위 스킬 발동 | Effect_Skill_Area | 플레이어 위치 | 0.5 | 스킬: 범위 |
| 플레이어 피격 | Effect_Hurt | 플레이어 위치 | 0.3 | |
| 몬스터 사망 | Effect_EnemyDeath | 몬스터 위치 | 0.5 | |
| 아이템 획득 | Effect_GetItem | 아이템 위치 | 0.3 | (선택) |

→ 이 표가 **이펙트 기획** 결과물입니다. 이걸 기준으로 **프리팹**을 만들고, **코드에서 생성 시점·위치**를 연결합니다.

# Part 6. 이펙트 자료구조 — 데이터로 어떻게 표현하는지

## 6.1 가장 단순한 구조: 이벤트별 프리팹 변수

**한 이벤트당 이펙트 프리팹 하나**를 변수로 두는 방식입니다.

```
[구조]
- hitEffectPrefab   (GameObject)  → 공격 타격 시 생성
- hurtEffectPrefab  (GameObject)  → 피격 시 생성
- deathEffectPrefab (GameObject)  → 몬스터 사망 시 생성
```

- **생성 시**: `Instantiate(hitEffectPrefab, 위치, 회전)`.  
- **자료구조**: “이벤트 → 프리팹”이 **변수 하나씩** 대응합니다.

## 6.2 스킬/공격과 묶는 구조: SkillData 안에 이펙트

스킬마다 **데미지, 쿨, 사운드, 이펙트**를 한 묶음으로 두는 방식입니다.

```
[구조: SkillData (ScriptableObject)]
- skillName
- damage, cooldown, range
- effectPrefab  (GameObject)  → 이 스킬 발동 시 생성할 이펙트
- soundClip    (AudioClip)   → 이 스킬 발동 시 재생할 소리
```

- **자료구조**: **스킬 하나 = 데이터 한 묶음**(ScriptableObject 한 개). 그 안에 **이펙트 프리팹 참조**가 들어 있습니다.  
- **장점**: “이 스킬에는 이 이펙트”가 **데이터만 바꿔서** 정해지고, 코드는 “스킬 데이터의 effectPrefab 사용”으로 통일할 수 있습니다.

## 6.3 이펙트 전용 데이터 타입 (선택)

이펙트만 따로 관리하고 싶을 때 쓰는 구조입니다.

```
[구조: EffectData (ScriptableObject)]
- effectId     (문자열)  → "hit_normal", "hurt"
- prefab       (GameObject)
- lifeTime     (숫자)    → 지속 시간(초)
```

- **자료구조**: “이펙트 ID + 프리팹 + 지속 시간”을 묶은 **타입** 하나.  
- 사용: 이벤트 발생 시 “effectId에 맞는 EffectData”를 찾아서, 그 prefab을 해당 위치에 Instantiate하고, lifeTime 후 제거.

# Part 7. 사운드 + 이펙트 통합 자료구조

## 7.1 “한 이벤트에 소리 + 이펙트”를 묶는 구조

**공격 타격**, **피격**처럼 “같은 순간에 **사운드도 재생**하고 **이펙트도 생성**”하는 경우, 두 가지를 **한 데이터**로 묶어 두면 기획·연결이 쉽습니다.

```
[구조: HitEventData (ScriptableObject) 예시]
- eventId       (문자열)   → "attack_hit", "hurt"
- soundClip     (AudioClip)  → 재생할 소리
- effectPrefab  (GameObject) → 생성할 이펙트
- effectLifeTime (숫자)      → 이펙트 지속 시간(선택)
```

- **자료구조**: “이벤트 ID + 사운드 클립 + 이펙트 프리팹 + (선택)지속 시간”을 묶은 **한 타입**.  
- **사용**: 공격이 닿을 때 “attack_hit용 HitEventData”를 읽어서 `Play(data.soundClip)`, `Instantiate(data.effectPrefab, 위치)`를 호출합니다.  
→ **기획표**의 “이 이벤트 → 이 사운드, 이 이펙트”가 **에셋 한 개**로 표현됩니다.

## 7.2 기획표와 자료구조 대응

| 기획표 (사운드·이펙트) | 자료구조로 표현 |
|------------------------|-----------------|
| 이벤트 이름 | eventId 또는 변수/필드 이름 |
| 재생할 클립 | AudioClip 타입 필드 (soundClip) |
| 생성할 이펙트 | GameObject 타입 필드 (effectPrefab) |
| 지속 시간 | float 타입 필드 (lifeTime) |
| 스킬별로 다르게 | SkillData 에셋마다 effectPrefab, soundClip 다르게 할당 |

→ **기획**에서 정한 **표 한 행**이, **코드/에셋**에서는 **필드 몇 개** 또는 **ScriptableObject 한 개**로 대응된다고 보면 됩니다.

| **사운드 자료구조** | 이벤트별 변수, 리스트+이름 매칭, ScriptableObject(이벤트 ID + clip) |
| **이펙트 기초** | GameObject, Prefab, Instantiate, Particle/Sprite, 지속 시간(lifeTime) |
| **이펙트 기획** | 이벤트별로 “무슨 프리팹을 어디에 생성할지” 표로 정리 |
| **이펙트 자료구조** | 이벤트별 프리팹 변수, SkillData 안의 effectPrefab, EffectData(ScriptableObject) |
| **통합 자료구조** | HitEventData 등으로 “사운드 + 이펙트”를 한 묶음으로 표현, 기획표 한 행 = 에셋/필드 한 세트 |

이 문서를 읽으면 **사운드·이펙트 기획**이 무엇인지, **자료구조**로 어떻게 표현하는지 **기초부터** 확인할 수 있습니다.  
연결·구현 방법은 `12-사운드-이펙트-연결방안.md`, 이펙트 구성은 15번 문서를 참고하세요.
