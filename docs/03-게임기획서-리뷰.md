Unity 로그라이크용 **예시 기획서 요약**입니다.  
각 항목이 **왜 있는지, 어떤 뜻인지**를 먼저 이해한 뒤, 팀 기획서(또는 첨부한 기획서)를 이 구조에 맞춰 대조하세요.

## 1. 게임기획서 리뷰 체크리스트

기획서는 “**이 게임이 뭔지, 무엇을 만들지**”를 팀이 공유하기 위한 문서입니다.  
체크리스트는 **그 문서에 꼭 있어야 할 내용**이 빠지지 않았는지 확인할 때 씁니다.  
기획서를 검토할 때 아래 항목을 하나씩 확인하세요.  
**담당자**와 **완료 여부**를 팀 단위로 기록하면 진행 상황 파악에 도움이 됩니다.

### 1.1 개요·목표

| 번호 | 확인 항목 | 담당 | ✓ |
|------|-----------|------|---|
| 1 | 게임 장르(로그라이크 등)가 명확히 정의되어 있는가? | | ☐ |
| 2 | 대상 유저(연령·플랫폼)가 기재되어 있는가? | | ☐ |
| 3 | 개발 목표(예: 8주 완성, MVP 범위)가 있는가? | | ☐ |
| 4 | 참고할 기존 게임 또는 레퍼런스가 정리되어 있는가? | | ☐ |

### 1.2 핵심 플레이

| 번호 | 확인 항목 | 담당 | ✓ |
|------|-----------|------|---|
| 5 | 플레이어가 하는 행동(이동, 공격, 아이템 사용 등)이 나열되어 있는가? | | ☐ |
| 6 | 조작 방법(키/버튼)이 구체적으로 적혀 있는가? | | ☐ |
| 7 | 승리 조건·패배 조건(게임 오버)이 있는가? | | ☐ |
| 8 | 한 판 플레이 타임(예: 5~15분) 추정이 있는가? | | ☐ |

### 1.3 콘텐츠·규칙

| 번호 | 확인 항목 | 담당 | ✓ |
|------|-----------|------|---|
| 9 | 스테이지/맵(플랫폼) 구성 방식이 설명되어 있는가? | | ☐ |
| 10 | 등장하는 적 종류와 행동이 정리되어 있는가? | | ☐ |
| 11 | 아이템·강화 요소가 정의되어 있는가? (없다면 “없음” 명시) | | ☐ |
| 12 | 난이도 조절 방안(체력, 적 수 등)이 언급되어 있는가? | | ☐ |

### 1.4 기술·리소스

| 번호 | 확인 항목 | 담당 | ✓ |
|------|-----------|------|---|
| 13 | 사용 엔진·플랫폼(Unity, PC 등)이 명시되어 있는가? | | ☐ |
| 14 | 필요한 에셋 종류(캐릭터, 맵, UI, 사운드)가 정리되어 있는가? | | ☐ |
| 15 | 팀 규모에 맞는 범위인가? (과도한 기능은 없는가?) | | ☐ |

### 1.5 리뷰 결과 정리

- **통과**: 위 항목 대부분 충족, 팀 합의 후 개발 진행  
- **수정 후 재리뷰**: 누락 항목 보완 후 다시 검토  
- **일정/범위 조정**: 기능을 줄이거나 단계를 나누어 재기획

## 2. 예시 게임기획서 요약 (2D 도트 액션 플랫포머)

아래는 **리뷰용 예시**로 쓸 수 있는 2D 도트 액션 플랫포머 게임 기획 요약입니다.  
(참고: 산나비, 셀레스테 등) 실제 팀 기획서는 이 구조를 참고해 더 구체적으로 작성하면 됩니다.

**개요**는 “이 게임의 이름, 장르, 누구를 위한 건지, 어떤 도구로 언제까지 만드는지”를 한눈에 보여 주는 부분입니다.  
여기가 명확해야 팀이 같은 목표를 공유하고, 에셋·코딩 담당이 작업 범위를 정할 수 있습니다.

### 2.1 개요

| 항목 | 내용 |
|------|------|
| **게임명** | (예) 플랫폼 점프 대작전 |
| **장르** | 2D 도트 액션 플랫포머 |
| **대상** | 초등학생·입문자 |
| **플랫폼** | PC (Windows/Mac) |
| **엔진** | Unity 6.3 LTS |
| **개발 기간** | 8~12주 (차시 기준) |

**핵심 플레이**는 “플레이어가 뭘 하고, 어떻게 조작하고, 언제 이기고 지는지”를 한 문장·한 줄씩 정리한 부분입니다.  
여기가 정해져 있어야 맵·몬스터·전투·UI를 모두 같은 방향으로 설계할 수 있습니다.

### 2.2 핵심 플레이

- **목표**: 스테이지를 진행하며 플랫폼을 점프하고 적을 처치해 목표 지점에 도달한다.
- **조작**: A/D(또는 방향키) 좌우 이동, 스페이스 점프, Z 공격.
- **승리**: (선택) 스테이지 목표 지점 도달 또는 보스 처치.  
- **패배**: 플레이어 HP 0 → 게임 오버 후 재시작.

**맵·던전**은 “맵이 랜덤인지, 방이 몇 개인지, 어떤 타일을 쓰는지”를 적는 부분입니다.  
이게 정해져야 맵 에셋·타일맵 양을 정할 수 있고, 개발 기간 안에 완성 가능한 수준으로 범위를 잡을 수 있습니다.

### 2.3 스테이지·맵(플랫폼)

- **스테이지**: 고정 맵 1~3개 (또는 소수 레벨).
- 플랫폼·바닥·벽 타일로 구성, 점프로 이동 가능한 구역.
- 타일: 바닥, 플랫폼, 벽, (선택) 다음 스테이지 입구.

**적**은 “몇 종이 나오는지, 어떻게 움직이고, 플레이어에게 어떻게 데미지를 주는지”를 적는 부분입니다.  
처음에는 1~2종만 정하고, “느리지만 튼튼”, “빠르지만 HP 낮음”처럼 한 줄 설명만 있어도 이후 데이터 테이블에서 수치를 채우기 좋습니다.

### 2.4 적(몬스터)

- 최소 1~2종: 슬라임(느림, 낮은 공격력), 박쥐(빠름, 낮은 HP) 등.
- 행동: 플레이어 발견 시 추적, 접촉 시 데미지.

**플레이어·전투**는 “플레이어에게 HP가 있는지, 공격 방식이 무엇인지, 몬스터 처치 후 보상이 있는지”를 적는 부분입니다.  
전투가 접촉인지 공격 버튼인지, 스킬이 있는지 한 줄만 정해 두면 구현 범위가 잡힙니다.

### 2.5 플레이어·전투

- 플레이어: HP, 좌우 이동·점프, 근접 공격(또는 단순 충돌 데미지).
- 적 처치 시 (선택) 코인·경험치·아이템 드롭.

**UI·피드백**은 “화면에 무엇을 보여 줄지”(HP, 점수, 게임 오버, 재시작 버튼 등)를 적는 부분입니다.  
필수인 것(HP, 게임 오버, 재시작)과 선택(점수, 층 수)을 구분해 두면 UI 문서(10)와 구현 범위를 맞추기 쉽습니다.

### 2.6 UI·피드백

- HP 바(또는 숫자), (선택) 점수·층 수, 게임 오버 화면, 재시작 버튼.

**리소스**는 “어떤 그래픽·사운드를 쓰는지, 1차와 2차 범위를 어떻게 나눌지”를 적는 부분입니다.  
플레이스홀더로 먼저 진행하고, 나중에 스프라이트·사운드로 교체하는 식으로 단계를 나누어 두면 팀 규모·기간에 맞게 완성할 수 있습니다. → [06 문서 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체)

### 2.7 리소스

- **1차**: 플레이스홀더(임시 스프라이트)로 진행 후 교체. 또는 간단 스프라이트만 사용. → [06 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체)
- **2차**: 2D 도트 스프라이트, 플랫폼 타일, 효과음·BGM 추가.

## 4. 기획서 리뷰 회의 시 참고

- **리뷰 전**: 기획서 초안을 팀원 모두가 미리 읽어 오기.
- **리뷰 중**: 체크리스트 항목별로 “있음/없음/수정 필요”만 빠르게 표시.

이렇게 하면 **게임기획서 리뷰 → 요구사항·시스템 설계 → 에셋·역할 분담 → 단계별 개발**까지 한 흐름으로 이어질 수 있습니다.
