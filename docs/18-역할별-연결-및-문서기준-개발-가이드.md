# 역할별 연결 및 문서 기준 개발

**기획·에셋 제작·프로그램(코딩)** 등 역할별 문서·역할은 **각각 분리**되어 있지만 **서로 연결**되어 확인할 수 있고,  
**기획 문서를 기준**으로 에셋 제작·구현이 이루어지며, **각 담당이 준비한 것**을 기준으로 동작·제작·구현이 이어집니다.  
기획 문서를 **언제든 보강**해도, **해당 문서를 기준**으로 개발·제작을 진행할 수 있습니다.

**연결 점검 시**: 수치·SO·프리팹은 **`_Content/`**, 규칙·씬전환 코드는 **`_Framework/`**, HUD는 **`_UI/`** 경계가 깨지지 않았는지 [README 세 폴더](../README.md#folder-layers)·[02](02-준비사항-체크리스트.md)·[06](06-팀원-역할-분담.md)·[README #folder-depth](../README.md#folder-depth)·[10 #ui-folder-check](10-UI-HUD-디자인-및-구현-가이드.md#ui-folder-check)로 확인합니다. **구조·코어·컨텐츠·UI** 네 축은 [README `#dev-flow-axes`](../README.md#dev-flow-axes). **트리거·인터랙션·이벤트** 는 [README `#interaction-events`](../README.md#interaction-events) · [README #folder-depth I1~I5](../README.md#folder-depth).

# Part 1. 역할별 문서·연결도

## 1.1 전체 흐름 (기획 → 에셋 → 코딩)

```
[ 기획 ]  ──산출물──►  [ 에셋 제작 ]  ──산출물──►  [ 프로그램·코딩 ]
   │                        │                          │
   │  GDD, 시스템 목록,      │  프리팹, 데이터 에셋,     │  스크립트, 씬,
   │  데이터·컨텐츠 테이블   │  스프라이트, 오디오       │  연결·동작 구현
   │                        │                          │
   └────────────────────────┴──────────────────────────┘
                    ▲
                    │  기준: 기획 문서
                    │  각 담당 준비물이 맞닿는 지점에서 "연결 확인"
```

- **기획**이 만든 **문서(산출물)** 가 **기준**이 됩니다.  
- **에셋 제작**은 기획의 **에셋 목록·데이터 테이블**을 보고 제작하고,  
- **코딩**은 기획의 **요구사항·시스템 설계·데이터 형식**을 보고 구현합니다.  
- 세 역할의 **준비물**이 **인터페이스**(형식·이름·참조 방법)로 맞닿아 있으면, **연결해서 확인**할 수 있습니다.

---

## 1.2 역할별 참고 문서 (분리·연결)

각 역할은 **자기 참고 문서**만 보고 작업하되, **다른 역할 산출물**과 **어디서 연결되는지** 아래 표로 확인할 수 있습니다.

| 역할 | 참고 문서 | 준비하는 산출물 | 연결되는 역할·문서 |
|------|------------------------|-----------------|---------------------|
| **기획** | 17, 03, 04, 05 (일부) | GDD, 시스템 목록, 데이터·컨텐츠 테이블, 요구사항 목록 | → 에셋: 05·08 / 코딩: 04·07 |
| **에셋 제작** (데이터·아트·사운드) | 05, 08, 11(맵 리소스), 15(이펙트) | 데이터 에셋, 프리팹, 스프라이트, 오디오 클립 | ← 기획 테이블 / → 코딩: Inspector 연결 |
| **프로그램·코딩** (게임로직·UI·시스템) | 04, 07, 08, 09, 10, 11, 12, 13, 14, 15, 16 | 스크립트, 씬, 프리팹 배치·연결 | ← 기획 04·07 / ← 에셋: 에셋 참조 |

- **확인 방법**: 기획 문서(테이블·요구사항 ID) ↔ 에셋 목록(05) ↔ 구현 단계(07)를 **한 줄로 추적**할 수 있도록, Part 3·4에서 **연결 확인 체크리스트**와 **기획 보강 시 갱신 절차**를 둡니다.

---

## 1.3 문서 기준 개발 원칙

- **기획 문서가 기준**  
  - “무엇을 만들지”, “수치는 무엇인지”, “어떤 시스템이 있는지”는 **기획 산출물(GDD, 테이블, 요구사항)** 에 정의합니다.  
  - 에셋 제작·코딩은 **이 문서를 보고** 제작·구현합니다.
- **준비된 것을 기준으로 동작**  
  - 기획에서 **테이블·요구사항**이 준비되면 → 에셋은 그 테이블에 맞는 **에셋·프리팹**을 준비하고,  
  - 에셋이 준비되면 → 코딩은 **그 에셋을 참조**해서 동작을 구현합니다.  
  - 각 단계는 “**앞 단계 준비물**”을 기준으로 작업하면, **연결 시 동작**이 맞습니다.
- **기획 보강 시**  
  - 기획 문서를 **수정·보강**하면, **변경된 부분**만 해당 문서(05, 08, 04, 07 등)와 대조하고,  
  - 해당하는 **에셋·구현**을 갱신한 뒤, Part 4의 **연결 확인**을 다시 하면 됩니다.

# Part 2. 기획 문서 기준 — 무엇을 참조하는지

## 2.1 에셋 제작이 참조하는 기획 문서

| 기획 산출물 | 에셋 담당이 하는 일 | 참고 문서 |
|-------------|----------------------|-------------|
| **데이터·컨텐츠 테이블** (17 Part 3) | 플레이어·몬스터·스킬·던전 수치 → ScriptableObject 에셋 생성, 필드명·타입 맞추기 | 05, 08 |
| **필요 에셋 목록** (05) | 캐릭터·맵·UI·이펙트·오디오 종류·개수 확인 후 제작 | 05, 11, 15 |
| **스킬 정의 테이블** | 스킬별 데미지·쿨·이펙트·사운드 → SkillData 에셋, 이펙트 프리팹 | 15 |
| **GDD 맵·몬스터 설명** | 맵 타일, 몬스터 외형 리소스 제작 범위 확인 | 05, 11 |

→ 에셋 제작은 **기획의 테이블·목록**을 기준으로 “무엇을, 몇 개, 어떤 형식으로” 만들지 정합니다.

## 2.2 코딩(구현)이 참조하는 기획 문서

| 기획 산출물 | 코딩 담당이 하는 일 | 참고 문서 |
|-------------|----------------------|-------------|
| **요구사항 목록** (04, 17 Part 4) | 요구사항 ID별로 구현할 기능 확인, Phase별 완료 조건 충족 | 04, 07 |
| **시스템(모듈) 목록·연결** (17 Part 2, 04) | 어떤 스크립트·모듈을 만들지, 누가 누구를 호출하는지 구현 | 04, 13 |
| **데이터 테이블·형식** (17 Part 3, 08) | PlayerStatsData, EnemyStatsData, SkillData 등 **형식**에 맞춰 참조만 사용 | 08, 13 |
| **GDD 핵심 플레이·조작** | 입력·전투·게임 오버 등 동작 규격 확인 | 03, 07 |

→ 코딩은 **기획의 요구사항·시스템·데이터 형식**을 기준으로 “무엇을 구현하고, 어떤 에셋을 참조할지” 정합니다.

---

## 2.3 기준 문서가 바뀌었을 때 (기획 보강)

- **기획 문서를 보강(수정)** 한 경우:
  1. **변경된 부분**을 명시 (예: “몬스터 테이블 1종 추가”, “스킬 1개 추가”).
  2. **에셋**: 변경된 테이블·목록에 맞춰 **새 에셋 추가** 또는 **기존 에셋 수정** (05, 08, 15 참고).
  3. **코딩**: 변경된 요구사항·시스템·데이터에 맞춰 **스크립트·연결 수정** (04, 07, 13 참고).
  4. **연결 확인**: Part 4 체크리스트로 “기획 ↔ 에셋 ↔ 코딩”이 다시 맞는지 확인.

→ **언제든 기획 문서를 보강**해도, **해당 문서를 기준**으로 위 순서만 따르면 **개발·제작**을 이어갈 수 있습니다.

# Part 3. 준비물 기준 동작 — 연결 확인

## 3.1 “준비된 것”이 맞닿는 지점 (인터페이스)

각 담당이 **준비한 것**이 아래 지점에서 맞닿을 때, **동작·제작·구현**이 이어집니다.

| 기획이 준비한 것 | 에셋이 준비한 것 | 코딩이 준비한 것 | 연결 확인 |
|------------------|------------------|-------------------|-----------|
| 플레이어 수치 테이블 | PlayerStatsData 에셋 (필드명·타입 일치) | PlayerController 등에서 stats 참조 | Inspector에서 에셋 연결 시 수치 반영 |
| 몬스터 수치 테이블 | EnemyStatsData 에셋 N개 | EnemyController에서 stats 참조, 스폰 시 할당 | 스폰 시 종류별 수치 동작 |
| 스킬 정의 테이블 | SkillData 에셋, 이펙트 프리팹, 사운드 클립 | PlayerCombat에서 skills 배열·DoSkill(s) | 스킬별 데미지·이펙트·사운드 동작 |
| 던전 설정 테이블 | (데이터 에셋 선택) | DungeonGenerator에서 설정 참조 | 맵 생성 규칙 반영 |
| 요구사항 ID·시스템 목록 | — | Phase별 구현·요구사항 충족 | 07 Phase 완료 조건·04 요구사항 체크 |

→ **준비물 기준 동작**: 기획 테이블 = 에셋 형식 = 코딩에서 참조하는 형식이 **같을 때** 연결이 되고, 동작이 보장됩니다.

## 3.2 역할별 “준비 완료” 확인 (자기 쪽)

| 역할 | 준비 완료 시 확인할 것 |
|------|------------------------|
| **기획** | GDD 초안, 시스템 목록, 플레이어·몬스터·스킬·던전(·아이템) 테이블, 요구사항 목록이 채워져 있고, 17 Part 5 검토 통과 |
| **에셋** | 05·기획 테이블에 있는 에셋이 생성되어 있고, **필드명·타입**이 기획·04·08에 정의된 형식과 같음 |
| **코딩** | 07 Phase별 완료 조건 충족, 스크립트가 **기획 데이터(에셋)** 만 참조하고, Inspector에서 에셋 연결 가능 |

# Part 4. 연결 확인 체크리스트 (전체 확인)

기획·에셋·코딩이 **모두 연결되어 있는지** 아래 순서로 확인할 수 있습니다.

## 4.1 기획 → 에셋 연결

- [ ] 기획 **데이터·컨텐츠 테이블**(17 Part 3)에 있는 **항목·수치**가, 에셋 **ScriptableObject**에 동일하게 반영되어 있다.
- [ ] 기획 **필요 에셋 목록**(05)에 있는 **프리팹·스프라이트·오디오**가, 프로젝트에 존재하거나 제작 예정으로 정리되어 있다.
- [ ] **스킬 정의 테이블**의 스킬 종류만큼 **SkillData** 에셋(및 필요한 이펙트·사운드)이 있다.

## 4.2 에셋 → 코딩 연결

- [ ] **데이터 에셋**(PlayerStatsData, EnemyStatsData, SkillData 등)의 **클래스명·필드명**이, 코딩 쪽 스크립트에서 참조하는 이름과 같다.
- [ ] **프리팹**(Player, Enemy, 이펙트 등)이 씬 또는 스크립트에서 **참조·Instantiate** 가능한 상태이다.
- [ ] **사운드·이펙트** 연결처(12)가 정해진 이벤트와 맞고, 해당 클립·프리팹이 할당 가능하다.

## 4.3 코딩 → 기획 연결

- [ ] **요구사항 ID**(04)별로 “해당 기능이 구현되었는지” Phase·스크립트와 대조할 수 있다.
- [ ] **시스템(모듈) 목록**(17 Part 2, 04)에 있는 모듈이, 스크립트·씬 구성으로 구현되었거나 예정으로 정리되어 있다.
- [ ] **기획 보강**으로 요구사항·테이블이 바뀌면, 변경분이 **에셋·코딩**에 반영·확인되었다.

## 4.4 통합 확인 (동작)

- [ ] **기획 문서**를 기준으로, **에셋**을 연결한 뒤 **실행**했을 때, 수치·스킬·맵 등이 기획 테이블·GDD와 같게 동작한다.
- [ ] **Phase별 완료 조건**(07)과 **요구사항**(04)을 기준으로, 필요한 기능이 구현·동작한다.

→ 위를 **역할별로 나누어** 확인해도 되고, **한 번에** “기획 ↔ 에셋 ↔ 코딩” 연결만 점검해도 됩니다.

# Part 5. 기획 보강 시 갱신·재확인 절차

기획 문서를 **언제든 보강**했을 때, **해당 문서 기준**으로 개발·제작이 계속 가능하도록 하는 절차입니다.

## 5.1 기획 문서 보강 시 할 일

1. **변경된 기획 문서**에 “변경 일자·변경 내용 요약”을 한 줄이라도 남긴다.  
   예: “몬스터 1종 추가(박쥐), 스킬 테이블에 범위 공격 추가.”
2. **영향받는 산출물**을 정리한다.  
   - 테이블 변경 → 에셋: 해당 데이터 에셋 추가·수정.  
   - 요구사항·시스템 변경 → 코딩: 해당 모듈·스크립트 추가·수정.
3. **에셋·코딩** 담당은 **변경된 부분**만 05, 08, 04, 07 등 **해당 문서**를 보고 제작·구현을 보강한다.
4. **연결 확인**(Part 4)을 **변경된 범위**에 대해 다시 실행한다.

## 5.2 문서 기준 개발 유지

- **기획 문서 = 단일 기준**  
  - “이 게임의 규칙·수치·시스템”은 **기획 산출물**에만 정의하고,  
  - 에셋·코딩은 **항상 기획 문서(최신)** 를 참조해서 제작·구현한다.
- **보강 후에도**  
  - “기획 문서를 보강했다” → “변경분만 에셋·코딩에 반영” → “연결 확인”  
  - 이 순서만 지키면 **문서 기준 개발**이 유지되고, **언제든 기획 문서를 기준으로** 구현·제작이 가능합니다.

위와 같이 진행하면 **기획·에셋 제작·코딩**이 **분리**되어 있으면서도 **연결**되어 확인할 수 있고, **기획 문서를 기준**으로 **준비된 것을 바탕**으로 동작·제작·구현이 이어지며, **기획 문서를 언제든 보강**해도 **해당 문서 기준**으로 개발·제작을 이어갈 수 있습니다.
