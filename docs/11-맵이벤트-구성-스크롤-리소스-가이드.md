# Part 1. 맵 이벤트

## 1.1 맵 이벤트란?

특정 **위치**나 **상황**에서 발생하는 동작입니다.  
예: 방 입장 시 몬스터 스폰, 특정 칸 밟을 때 함정·아이템, 문 열기, 다음 층 입구 등.

---

## 1.2 구현 방안 비교

| 방안 | 난이도 | 장점 | 단점 | 추천 |
|------|--------|------|------|:----:|
| **A. 그리드 번호 + 이벤트 테이블** | ★★☆ | 맵 데이터와 이벤트를 숫자로 매칭 | 이벤트 종류 늘면 테이블 관리 필요 | **가장 추천** |
| **B. 트리거 오브젝트** | ★★☆ | 씬에 오브젝트 배치로 직관적 | 오브젝트 수 많아지면 무거움 | 소수 구역만 이벤트 시 |
| **C. 이벤트 ScriptableObject** | ★★★ | 이벤트 종류·설정 데이터화 | 설계·코드량 증가 | 이벤트 종류 많을 때 |

---

## 1.3 방안 A: 그리드 번호 + 이벤트 테이블 (가장 쉬움 ★)

**방식**: 맵 그리드에 **이벤트용 번호**를 부여하고, 플레이어/오브젝트가 그 칸에 들어오면 해당 번호에 맞는 동작 실행.

### (1) 그리드 확장 (칸 의미 추가)

```
0 = 빈칸  1 = 바닥  2 = 벽  3 = 이벤트_몬스터스폰  4 = 이벤트_다음층입구  5 = 함정
```

### (2) 이벤트 실행 (플레이어 위치 기반)

```csharp
// 플레이어가 이동한 뒤, 현재 그리드 칸 확인
void CheckMapEvent(int gridX, int gridY)
{
    int cell = map.grid[gridX, gridY];
    switch (cell)
    {
        case 3: SpawnEnemiesInCurrentRoom(); break;
        case 4: GoToNextFloor(); break;
        case 5: ApplyTrapDamage(); break;
    }
}
```

- **플레이어 그리드 좌표**: 월드 좌표를 `(int)(pos.x / cellSize)`, `(int)(pos.y / cellSize)` 등으로 칸 좌표로 변환한 뒤 `CheckMapEvent` 호출.

### (3) 1회만 실행하려면

- `bool[,] eventTriggered` 배열을 두고, 이벤트 실행 시 `eventTriggered[x,y] = true`로 표시.  
- `CheckMapEvent`에서 `if (eventTriggered[gridX, gridY]) return;` 처리하면 한 번만 실행.

---

## 1.4 방안 B: 트리거 오브젝트 (구역 진입 시)

**방식**: 빈 오브젝트 + Collider(Is Trigger) + 스크립트. 플레이어가 들어오면 `OnTriggerEnter`에서 이벤트 실행.

```csharp
// MapEventTrigger.cs — 트리거 오브젝트에 부착
void OnTriggerEnter(Collider other)
{
    if (other.CompareTag("Player"))
        OnPlayerEnter();  // 몬스터 스폰, 문 열기 등
}
```

- **장점**: 에디터에서 이벤트 구역 크기·위치를 시각적으로 배치 가능.  
- **용도**: “이 방 전체 진입 시”, “이 문 앞 도달 시” 등 **구역 단위** 이벤트에 적합.

---

## 1.5 맵 이벤트 구현 단계 요약

| 단계 | 할 일 |
|------|--------|
| 1 | 그리드 칸에 이벤트 번호(3, 4, 5…) 부여하거나, 트리거 오브젝트 배치 |
| 2 | 플레이어 위치 → 그리드 좌표 변환 후 해당 칸 번호 확인, 또는 OnTriggerEnter 수신 |
| 3 | 번호/트리거별로 실행할 동작(스폰, 씬 전환, 데미지 등) 구현 |
| 4 | (선택) 1회만 실행 시 `eventTriggered` 플래그 또는 트리거 비활성화 |

# Part 2. 맵 구성

## 2.1 맵 구성이란?

맵의 **구조**를 어떻게 나누고, 어떤 **요소(방·복도·층·타일 종류)** 로 채울지 설계하는 것입니다.

---

## 2.2 구성 요소 정리

| 요소 | 설명 | 데이터 표현 예시 |
|------|------|-------------------|
| **방(Room)** | 사각형 구역, 바닥+벽 | 그리드에서 (x,y,w,h) 영역, 칸 값 1·2 |
| **복도(Corridor)** | 방과 방을 잇는 통로 | 1칸 너비 줄, 칸 값 1 |
| **층(Floor)** | 1층·2층… 별도 맵 또는 같은 그리드 확장 | 씬 전환 또는 그리드 z/레이어 |
| **타일 종류** | 바닥/벽/물/함정 등 | grid 값 0,1,2,5… |

---

## 2.3 맵 구성 설계 (가장 쉬운 순서)

### (1) 1층만 있을 때

1. **그리드 크기** 정하기 (예: 40×30).
2. **방 리스트**: 각 방 = (시작X, 시작Y, 가로, 세로). 랜덤 또는 고정.
3. **방 채우기**: 각 방 영역에 가장자리=벽(2), 안쪽=바닥(1). (기존 `MakeSingleRoom` 활용.)
4. **복도**: 두 방의 중심(또는 가장자리)을 잇는 직선/L자 구간을 1(바닥)로 채우기.
5. **이벤트 칸**: 다음 층 입구, 스폰 방 등 원하는 칸에 3·4·5 등 부여.

### (2) 여러 층일 때 (가장 쉬운 방식)

- **방식 1**: 층마다 **별도 씬**. 다음 층 입구(그리드 4) 도달 시 `SceneManager.LoadScene("Floor2")` 등으로 씬 전환 후 새 맵 생성.
- **방식 2**: **같은 씬**, 그리드만 다시 생성. “다음 층 입구” 이벤트에서 `DungeonGenerator.Generate()` 다시 호출 후 플레이어 위치만 시작 방으로 옮기기.

---

## 2.4 맵 구성 데이터 (ScriptableObject 예시)

```csharp
// DungeonSettingsData에 방·복도 규칙만 두고, 생성 알고리즘은 DungeonGenerator가 수행
[CreateAssetMenu(menuName = "Roguelike/Dungeon Settings")]
public class DungeonSettingsData : ScriptableObject
{
    public int width = 40, height = 30;
    public int roomCountMin = 5, roomCountMax = 9;
    public int roomSizeMin = 4, roomSizeMax = 8;
    public int corridorWidth = 1;
}
```

- **맵 구성** = 위 설정 + `DungeonGenerator`에서 방 배치·복도 연결 순서를 코드로 구현.  
  자세한 알고리즘은 08 문서, 07 문서 Phase 2 참고.

---

## 2.5 맵 구성 단계 요약

| 단계 | 할 일 |
|------|--------|
| 1 | 그리드 크기·방 개수·방 크기 범위 정하기 |
| 2 | 방 사각형 생성(겹치지 않게) → 각 방을 벽/바닥으로 채우기 |
| 3 | 방끼리 복도로 연결(직선/L자로 1로 채우기) |
| 4 | (선택) 이벤트 칸(다음 층 입구 등) 배치, 층 전환 시 새 맵 생성 또는 씬 전환 |

---

## 2.6 맵·레벨 형태별 만드는 방법

2D 플랫포머에서 자주 쓰는 **레벨 형태**별로, 데이터 정의·만드는 순서·구현 요점을 정리했습니다.  
기본 그리드(0=빈칸, 1=바닥, 2=벽)는 08 문서를 참고하고, 여기서는 **형태별 차이**만 집중합니다.

### 형태 요약 표

| 형태 | 설명 | 그리드/데이터 | 난이도 |
|------|------|----------------|:------:|
| **선형(가로 스크롤)** | 왼쪽→오른쪽 한 방향, 플랫폼이 나열 | 가로로 긴 1층, 플랫폼 높이만 다름 | ★☆☆ |
| **수직(클라이밍)** | 아래→위로 오르는 맵 | 세로로 긴 1층, 발판·사다리 | ★★☆ |
| **방형(룸 연결)** | 방 여러 개가 복도로 연결 | 방 (x,y,w,h) + 복도 | ★★☆ |
| **분기형** | 갈림길·되돌아오기 구간 있음 | 선형+분기점 또는 방형 확장 | ★★★ |
| **아레나형** | 넓은 한 구역에서 전투·이벤트 | 큰 방 하나, 이벤트 칸만 추가 | ★☆☆ |

---

### (1) 선형(가로 스크롤) 레벨

**특징**: 왼쪽에서 오른쪽으로 진행. 플랫폼이 **가로 한 줄** 또는 **여러 높이**로 나열. 산나비·마리오 같은 기본 스테이지에 가깝습니다.

**그리드 정의**

- **가로는 넉넉히** (예: 60~120칸), **세로는 1~3줄** 바닥 또는 플랫폼 높이만 다르게.
- 칸 의미: `0=빈칸(낙사)`, `1=바닥/플랫폼`, `2=벽`.

**만드는 순서**

1. **그리드 크기** 정하기: 예) `width=80`, `height=15`. (세로 15면 플랫폼을 여러 높이에 둘 수 있음.)
2. **바닥 한 줄** 채우기: `y=0`(또는 1) 전 구간을 1(바닥)로, 왼쪽/오른쪽 끝은 2(벽).
3. **플랫폼 추가**: 중간중간 `y=2`, `y=4` 등에서 **가로로 1**을 몇 칸씩 넣어 “띄어난 발판” 만들기.
4. **구멍**: 바닥 줄에서 일부 칸을 0으로 두면 낙사 구역.
5. **(선택)** 이벤트 칸: 목표 지점·체크포인트에 3·4 번호 부여.

**의사 코드**

```csharp
void MakeLinearLevel(SimpleMapData map)
{
    int h = map.height - 1;  // 바닥 줄
    for (int x = 0; x < map.width; x++)
        map.grid[x, h] = (x == 0 || x == map.width - 1) ? 2 : 1;  // 양끝 벽, 나머지 바닥

    // 플랫폼: y= h-3 에서 x= 10~15 구간만 바닥
    for (int x = 10; x <= 15; x++)
        map.grid[x, h - 3] = 1;
    // 같은 방식으로 다른 높이에 플랫폼 추가
}
```

**Unity 적용**: Tilemap이면 위 그리드대로 `SetTile` 호출. 프리팹이면 08 문서의 “그리드 → Instantiate” 이중 for문 사용. **카메라**는 플레이어 X만 따라가게 하면 가로 스크롤됩니다.

---

### (2) 수직(클라이밍) 레벨

**특징**: 아래에서 위로 올라가는 맵. **세로가 긴** 한 맵, 발판·사다리·점프 구간으로 구성.

**그리드 정의**

- **세로를 넉넉히** (예: height=60), 가로는 좁게 (예: width=10~20).
- 칸 의미: `0=빈칸`, `1=바닥/발판`, `2=벽`, (선택) `6=사다리(올라갈 수 있는 칸)`.

**만드는 순서**

1. **그리드 크기**: 예) `width=16`, `height=80`.
2. **양쪽 벽**: `x=0`, `x=width-1` 전 구간을 2(벽)로.
3. **발판**: 중간 x(예: 3~12)에서 **세로로 떨어져 있는** 작은 플랫폼(1)을 여러 개. y를 5, 12, 20, 28… 처럼 간격 두고 가로 3~5칸씩 1로 채우기.
4. **(선택)** 사다리: 특정 x 구간을 6으로 두고, 플레이어가 그 칸에 있으면 위/아래 이동 허용.
5. **시작/목표**: 아래쪽 한 칸=스폰, 맨 위 한 칸=클리어 이벤트(4).

**의사 코드**

```csharp
void MakeVerticalLevel(SimpleMapData map)
{
    int w = map.width;
    for (int y = 0; y < map.height; y++) {
        map.grid[0, y] = 2;
        map.grid[w - 1, y] = 2;
    }
    // 발판: y= 5, 12, 20, 28 ... 에서 x= 3~7 만 바닥
    int[] platformY = { 5, 12, 20, 28, 35, 42, 50 };
    foreach (int py in platformY)
        for (int x = 3; x <= 7; x++)
            map.grid[x, py] = 1;
}
```

**Unity 적용**: 카메라가 **플레이어 Y**를 따라가도록 하면 세로 스크롤. 2D에서는 Y 위로 올라가게 하려면 “위쪽이 +Y”인지 “아래쪽이 +Y”인지 팀 규칙 통일.

---

### (3) 방형(룸 연결) 레벨

**특징**: **방** 여러 개를 **복도**로 이어서 탐험. 08·위 2.3절의 “방 채우기 + 복도 연결”과 동일합니다.

**그리드 정의**

- 칸 의미: `0=빈칸`, `1=바닥`, `2=벽`, `3=이벤트`, `4=다음 층 입구` 등.
- **방 리스트**: 각 방 = (시작X, 시작Y, 가로, 세로). 겹치지 않게 배치.

**만드는 순서**

1. 그리드 크기·방 개수·방 크기 범위 정하기 (예: 40×30, 방 5~9개, 방 4~8칸).
2. **방 위치 랜덤(또는 고정)**: 각 방 (x,y,w,h) 정한 뒤, `MakeSingleRoom(map, x, y, w, h)`로 벽·바닥 채우기.
3. **복도**: 두 방의 중심(또는 가장자리)을 잇는 직선/L자 구간을 1(바닥)로 채우기. 복도 너비 1칸이면 한 줄만 1로.
4. **이벤트**: 시작 방 한 칸, 다음 층 입구(4), 몬스터 스폰 방(3) 등 부여.
5. (선택) **문**: 방과 복도 사이를 2(벽)로 막아 두었다가 이벤트로 1로 바꾸면 “문 열림” 연출 가능.

**구현 참고**: 08 문서 “방 하나 채우기” + “방 여러 개 + 복도 연결” 절을 그대로 사용하면 됩니다.

---

### (4) 분기형 레벨

**특징**: **갈림길**이 있고, 한쪽은 숨은 구역·아이템, 한쪽은 목표로 직행. 되돌아오는 구간이 있을 수 있음.

**그리드 정의**

- **선형을 베이스**로 두고, **분기점**에서 위/아래 또는 좌측 작은 방으로 뻗는 “가지”를 추가.
- 또는 **방형**에서 방 개수를 늘리고, 일부 방은 “선택 진입”으로 두면 분기처럼 쓸 수 있음.

**만드는 순서**

1. **메인 경로** 먼저: 선형처럼 왼쪽→오른쪽 한 줄(또는 한 줄 + 플랫폼)을 1·2로 채움.
2. **분기점** 정하기: 특정 x(예: 30, 50)에서 위쪽 또는 아래쪽으로 1(바닥)을 몇 칸 연속해서 “가지” 만듦.
3. **가지 끝**에 작은 방 또는 이벤트 칸(보상·몬스터) 배치.
4. 되돌아오는 길은 **같은 가지**를 역으로 오면 되므로, 가지를 1로만 채워 두면 됨.
5. **(선택)** 가지를 “한 번만 들어갈 수 있는” 이벤트(문 잠금)로 만들려면, 이벤트 번호 + `eventTriggered`로 1회만 열리게 처리.

**의사 코드**

```csharp
// 선형 바닥 만든 뒤, x=40 에서 위쪽으로 5칸 가지 추가
void AddBranch(SimpleMapData map, int branchX, int branchLength)
{
    int floorY = GetFloorYAt(branchX);  // 그 x에서 바닥 y
    for (int i = 1; i <= branchLength; i++)
        map.grid[branchX, floorY + i] = 1;  // 위로 1 채우기
}
```

---

### (5) 아레나형 레벨

**특징**: **넓은 한 구역** 안에서 전투·이벤트. 복도 없이 “방 하나”가 전부. 보스전·서바이벌 구역에 적합.

**그리드 정의**

- **방 하나** = 맵 전체 또는 그리드 대부분. (예: 30×20 한 방.)
- 칸 의미: `0=빈칸(밖)`, `1=바닥`, `2=벽`, `3=스폰`, `4=목표/보스 트리거` 등.

**만드는 순서**

1. **한 방**만 만들기: `MakeSingleRoom(map, 0, 0, map.width, map.height)` 로 전체를 한 방으로 채움.
2. **장애물**(선택): 안쪽 일부를 2(벽)로 막아 “엄폐물” 만들기.
3. **스폰 위치**: 플레이어는 (1,1), 적은 (width-2, height-2) 등 고정 또는 리스트로 여러 개.
4. **이벤트**: “적 전멸 시 문 열림”이면 특정 칸을 2로 막아 두었다가, 이벤트에서 1로 바꾸기.
5. **(선택)** 중앙에 “보스 등장” 트리거 칸(4) 두고, 밟으면 보스 스폰 + 카메라 고정.

**Unity 적용**: 카메라는 “방 안만 비추기”로 **맵 경계(Clamp)** 를 이 방 크기로 두면, 아레나 밖으로 나가지 않게 할 수 있음.

---

### 형태별 선택 요약

| 만들 레벨 | 추천 형태 | 참고 절 |
|-----------|-----------|---------|
| 스테이지 1·2처럼 왼→오 진행 | 선형(가로 스크롤) | (1) |
| 올라가는 탑·산 | 수직(클라이밍) | (2) |
| 던전·여러 방 탐험 | 방형(룸 연결) | (3) |
| 갈림길·숨은 방 | 분기형 | (4) |
| 보스전·한 방 전투 | 아레나형 | (5) |

위 순서대로 **그리드 채우기 → 08 문서의 배치 로직(Tilemap/프리팹)** 으로 그리면, 형태별 맵을 같은 데이터 구조로 만들 수 있습니다.

# Part 3. 맵 스크롤

## 3.1 맵 스크롤이란?

화면에 보이는 영역이 **움직이는** 효과입니다.  
- **카메라가 플레이어를 따라감** → 플레이어 기준으로 맵이 스크롤되는 것처럼 보임.  
- **맵(또는 배경)을 직접 이동** → 카메라 고정, 맵만 움직이게 할 때도 “맵 스크롤”이라고 부를 수 있음.

---

## 3.2 구현 방안 비교

| 방안 | 난이도 | 장점 | 단점 | 추천 |
|------|--------|------|------|:----:|
| **A. 카메라가 플레이어 추적** | ★☆☆ | 구현 간단, 2D에 적합 | 맵이 화면보다 작으면 의미 적음 | **가장 추천** |
| **B. Cinemachine** | ★★☆ | 부드러운 추적·영역 제한 쉬움 | 패키지 설치·설정 필요 | 2D 풀 quality 시 |
| **C. 맵(타일맵) 직접 이동** | ★★☆ | 카메라 고정 가능 | 맵 전체가 움직여야 해서 구조 복잡 | 특수한 연출 시 |

---

## 3.3 방안 A: 카메라가 플레이어 추적 (가장 쉬움 ★)

**방식**: 매 프레임(또는 LateUpdate)에서 카메라 위치를 플레이어 위치와 동기화.  
2D: X,Y만 맞추고 Z(깊이)는 offset으로 고정.

```csharp
// SimpleCameraFollow.cs — Main Camera에 부착
using UnityEngine;

public class SimpleCameraFollow : MonoBehaviour
{
    public Transform target;  // 플레이어
    public Vector3 offset = new Vector3(0, 0, -10);  // 2D: z=-10 등
    public float smoothSpeed = 5f;  // 0이면 즉시 따라감

    void LateUpdate()
    {
        if (target == null) return;
        Vector3 desired = target.position + offset;
        if (smoothSpeed <= 0)
            transform.position = desired;
        else
            transform.position = Vector3.Lerp(transform.position, desired, smoothSpeed * Time.deltaTime);
    }
}
```

- **2D**: Camera는 Orthographic, offset.z = -10 등으로 고정.  
- **맵이 화면보다 클 때**: 이렇게만 해도 “맵 스크롤”처럼 보입니다.  
- **맵 경계**: 맵 최소/최대 x,y를 두고, `desired.x = Mathf.Clamp(desired.x, minX, maxX)` 등으로 카메라가 맵 밖으로 나가지 않게 할 수 있음.

---

## 3.4 방안 B: Cinemachine (선택)

- **Virtual Camera** 생성 후 Follow에 플레이어 할당.  
- 맵 경계는 **CinemachineConfiner** + **Collider**로 제한 가능.  
- “가장 쉬운 스크롤”만 필요하면 **방안 A**로 충분합니다.

---

## 3.5 맵 스크롤 단계 요약

| 단계 | 할 일 |
|------|--------|
| 1 | 카메라에 플레이어 추적 스크립트 추가, target에 플레이어 할당 |
| 2 | 2D: offset.z 고정 등 필요한 축만 사용 |
| 3 | (선택) 맵 경계 안에서만 카메라가 움직이도록 Clamp 적용 |

# Part 4. 맵 리소스 제작

## 4.1 맵 리소스란?

맵을 그리는데 쓰는 **타일 이미지**, **프리팹**, **타일맵 에셋** 등입니다.

---

## 4.2 제작 방안 비교

리소스가 없을 때는 프로그래머가 **플레이스홀더로 먼저 작업한 뒤 나중에 교체**하는 방식으로 진행합니다. → [06 문서 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체)

| 방안 | 난이도 | 장점 | 단점 | 추천 |
|------|--------|------|------|:----:|
| **A. 단일 스프라이트 타일** | ★★☆ | PNG 1장씩 배치, 이해하기 쉬움 | 타일 많으면 draw call 증가 | **가장 추천** |
| **B. Tilemap + Tile Palette** | ★★☆ | Unity 2D 표준, 타일 끼워 넣기 편함 | 2D 전용 | 2D 픽셀 |
| **C. 타일셋(아틀라스)** | ★★☆ | 한 장에 여러 타일, 메모리 효율 | 슬라이스·설정 필요 | 타일 종류 많을 때 |

---

## 4.3 방안 A: 단일 스프라이트 타일 (가장 쉬운 이미지 사용)

### (1) 제작 규칙

- **타일 1종 = PNG 1장** (또는 여러 장).  
- **크기 통일**: 16×16, 32×32, 64×64 등. 맵에서 “1칸 = 1유닛”이면 Pixels Per Unit을 그에 맞춤(예: 32).

### (2) Unity 설정

1. PNG를 Project에 넣음.  
2. Texture Import: **Sprite (2D and UI)**, Pixels Per Unit = 32(또는 타일 한 칸 크기).  
3. **Mesh Type**: Full Rect.  
4. 벽/바닥별로 **프리팹** 생성: 빈 오브젝트 + SpriteRenderer, Sprite에 해당 스프라이트 할당.  
5. 맵 생성 시 `Instantiate(wallTilePrefab, position, Quaternion.identity)` 등으로 배치. (기존 그리드 배치와 동일.)

### (3) 리소스 목록 예시

| 리소스 | 용도 | 크기(권장) |
|--------|------|------------|
| floor.png | 바닥 타일 | 32×32 |
| wall.png | 벽 타일 | 32×32 |
| (선택) water.png | 물·장애 | 32×32 |

---

## 4.4 방안 B: Tilemap + Tile Palette (2D)

- **Tilemap** 오브젝트 생성 (Grid 하위).  
- **Tile Asset**: 스프라이트를 드래그해 Tile 에셋 생성.  
- **Tile Palette** 창에서 Tile을 팔레트에 올리고, 브러시로 맵에 그리거나, 스크립트에서 `Tilemap.SetTile(cellPos, tile)` 로 배치.  
- **맵 리소스** = **Tile 에셋**(스프라이트 기반) + **팔레트**. 랜덤 던전은 그리드 데이터 생성 후 `SetTile`로 채우면 됨.

---

## 4.5 방안 C: 타일셋(아틀라스)

- **한 PNG에 여러 타일**을 격자로 배치 (예: 4×4 타일, 각 32×32).  
- Unity에서 **Sprite Mode = Multiple**, **Sprite Editor**로 슬라이스.  
- 각 슬라이스를 Tile로 쓰거나, 프리팹의 Sprite로 지정.  
- **장점**: 텍스처 수 감소. **단점**: 슬라이스·이름 관리 필요.

---

## 4.6 맵 리소스 제작 단계 요약

| 단계 | 할 일 |
|------|--------|
| 1 | 방식 선택: 스프라이트 타일 또는 Tilemap. 리소스 없으면 [06 §1.6](06-팀원-역할-분담.md#16-리소스-없을-때-플레이스홀더--교체) 플레이스홀더→교체 |
