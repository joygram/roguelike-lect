# 코드 구조 설계 — 세 폴더와 이벤트

게임이 복잡해질수록 코드가 뒤엉키는 문제가 생긴다. 파일 수가 늘어날수록 "어디에 무엇을 두는가"를 처음부터 정해 두는 것이 중요하다. 이 문서는 그 규칙을 설명한다.

00~06 기본 강좌를 완성한 뒤, 코드를 더 체계적으로 정리하고 싶을 때 읽으면 된다.

---

## 세 폴더로 코드 나누기

프로젝트가 커지면 스크립트를 세 종류로 나누는 것이 유지보수에 유리하다.

| 폴더 | 역할 | 넣는 것 | 넣으면 안 되는 것 |
|------|------|---------|-----------------|
| `Assets/_Framework/` | **규칙·흐름** — 게임의 법칙만 담당 | 씬 전환, HP/사망 판정, 이벤트 채널 | 특정 적 수치, UI 문자열, Canvas 직접 조작 |
| `Assets/_Content/` | **내용물** — 실제 게임 데이터 | ScriptableObject 에셋, 적·맵 프리팹, 스테이지 씬 | `FindObjectOfType<UIManager>()` 같은 UI 직접 참조 |
| `Assets/_UI/` | **표시·입력** — 화면에 보이는 것만 담당 | Canvas, HUD, 패널, 버튼 | `currentHp -= damage` 같은 게임 로직 |

**연결 원칙:**
- UI는 `_Framework`의 이벤트만 구독한다
- `_Content`는 `_Framework`에 데이터를 주입한다
- `_UI`와 `_Content`는 서로 직접 참조하지 않는다

### 왜 이렇게 나누는가

HP 바 UI가 직접 `playerHealth.TakeDamage()`를 호출하면, HP 바를 바꿀 때 전투 코드도 수정해야 한다. 반대로, 전투 코드가 HP를 이벤트로 알리고 HP 바가 그 이벤트를 구독한다면 — 둘 중 하나만 바꿔도 다른 쪽이 영향을 받지 않는다.

규모가 작을 때는 차이가 없어 보이지만, 스테이지가 늘어나고 적 종류가 많아질수록 이 구분이 개발 속도를 지킨다.

## 이벤트 시스템 — `GameEvents`

`_Framework/`에 이벤트 채널을 하나 만들어 두면, 여러 스크립트가 서로를 직접 참조하지 않아도 통신할 수 있다.

```csharp
// Assets/_Framework/Runtime/GameEvents.cs
using System;

public static class GameEvents
{
    public static event Action<int, int> PlayerHpChanged;  // (현재 HP, 최대 HP)
    public static event Action GameOver;
    public static event Action StageCleared;

    public static void RaisePlayerHp(int current, int max) => PlayerHpChanged?.Invoke(current, max);
    public static void RaiseGameOver() => GameOver?.Invoke();
    public static void RaiseStageCleared() => StageCleared?.Invoke();
}
```

**사용 방법:**

HP가 바뀌는 곳(전투 코드)에서:
```csharp
// PlayerHealth.cs
_hp -= amount;
GameEvents.RaisePlayerHp(_hp, maxHp);  // 이벤트 발행
```

HP 바 UI에서:
```csharp
// HpBarUI.cs
void OnEnable()
{
    GameEvents.PlayerHpChanged += OnHpChanged;
}

void OnDisable()
{
    GameEvents.PlayerHpChanged -= OnHpChanged;  // 메모리 누수 방지
}

void OnHpChanged(int current, int max)
{
    fillImage.fillAmount = (float)current / max;
}
```

이렇게 하면 `HpBarUI`는 `PlayerHealth`를 전혀 몰라도 된다. HP가 바뀌는 시점에 이벤트만 발행되면 HP 바가 자동으로 반응한다.

## 씬 전환 — `SceneFlow`

씬을 이동하는 코드를 `_Framework/`에 한 곳에 모아둔다. UI 버튼은 이 메서드만 호출한다.

```csharp
// Assets/_Framework/Runtime/SceneFlow.cs
using UnityEngine;
using UnityEngine.SceneManagement;

public class SceneFlow : MonoBehaviour
{
    public void GoToMainMenu() => SceneManager.LoadScene("MainMenu");
    public void GoToGame()     => SceneManager.LoadScene("Game");
    public void ReloadStage()  => SceneManager.LoadScene(SceneManager.GetActiveScene().name);
}
```

씬 이름 문자열(`"MainMenu"`, `"Game"`)은 `_Content/`의 ScriptableObject에서 관리하면 씬 이름을 바꿔도 코드 수정이 한 곳에서만 일어난다.

**Build Settings 등록 순서 (File → Build Settings):**

| Index | 씬 |
|:-----:|-----|
| 0 | `MainMenu` — 빌드 시 첫 화면 |
| 1 | `Game` |
| 2~ | `Stage_1_1`, `Stage_1_2`, … (스테이지가 여러 개일 때) |

## 구현 순서

세 폴더 구조로 작업할 때의 권장 순서:

| 순서 | 대상 | 이유 |
|:----:|------|------|
| 1 | `_Framework/` — 이벤트 채널, 씬 전환 | 없으면 UI와 컨텐츠가 뒤엉킴 |
| 2 | `_Content/` — 최소 1개 스테이지, 1종 적 | 기반에 붙일 실제 데이터 |
| 3 | `_UI/` — HUD, 게임오버 패널 | 이벤트 채널이 완성된 후 구독 |
| 4 | `_Content/` 확장 | 기반·UI 수정 없이 씬·SO만 추가 |

## 경계 위반 예시와 수정 방법

| 잘못된 예 | 올바른 방향 |
|-----------|-------------|
| `HpBarUI`에서 `TakeDamage()` 호출 | 데미지는 전투 코드만; UI는 이벤트로 HP를 받아 표시만 |
| `UIManager`에 스테이지 클리어 조건 | 컨텐츠/기반에서 조건 판정 후 `GameEvents.RaiseStageCleared()` |
| `PlayerController`에서 `Find("Canvas")` | 기반은 UI를 모른다; 이벤트로 통신 |
| 보스 패턴 수치가 코드 상수 | `BossData` ScriptableObject에 (`_Content/`) |
